# ADR-0006: Стратегия рендеринга — RSC-first, ISR с cache tags для каталога

- **Статус:** принят
- **Дата:** 2026-07-03
- **Авторы:** Engineering (по итогам аудитов, отчёты 03–04)

## Контекст

33 из 40 страниц — целиком клиентские: SEO-контент рендерится в браузере (Google получает пустую разметку), данные грузятся водопадом `useEffect`, middleware выполняет Supabase-auth на каждый запрос включая статику. Модель рендеринга инвертирована: дорогое делается всегда, дешёвое — никогда.

## Решение

1. Страница — Server Component по умолчанию; `"use client"` — только листовые интерактивные островки (кнопка «в корзину», избранное, переключатели).
2. Классы маршрутов и их стратегия:
   - **Каталог** (товар, категория, бренд, главная, popular/specials): ISR + cache tags (`product:{id}`, `category:{slug}`); ревалидация — `revalidateTag` при изменении данных.
   - **Контент** (блог, доставка, юридические): static.
   - **Персональное** (account, cart, checkout, admin): dynamic.
3. Middleware-auth — только на матчерах `/account`, `/admin`, `/checkout`.
4. Каждый новый маршрут обязан объявить свой класс в PR (Playbook §3 App Router).

Миграция — постранично (blueprint D1), начиная с контентных страниц.

## Последствия

- Проще: SEO-контент в HTML; TTFB/LCP из CDN-кэша; БД перестаёт быть на пути каждого просмотра; нагрузка и стоимость падают.
- Сложнее: дисциплина инвалидации (изменил товар — ревалидируй tag); интерактив выделяется в островки; отладка кэша требует внимания.
- Затронут Playbook §1.4, §8.

## Альтернативы

- **Оставить CSR + prerender для ботов** — отклонено: два пути рендеринга, cloaking-риски, не решает нагрузку.
- **Полный SSG на билд** — отклонено: не масштабируется на растущий каталог (пересборка всего на любое изменение).
- **SSR без кэша (всё dynamic)** — отклонено: платить БД-запросом за каждый просмотр каталога бессмысленно — каталог одинаков для всех.
