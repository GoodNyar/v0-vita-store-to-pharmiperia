# ADR-0001: Единый источник данных каталога — Supabase (Postgres)

- **Статус:** принят
- **Дата:** 2026-07-03
- **Авторы:** Engineering (по итогам аудитов, отчёты 02–03)

## Контекст

В проекте исторически два несовместимых источника данных о товарах: хардкод `lib/data.ts` (16 товаров, числовые id, один язык) и Supabase-схема (UUID, `name_ru/name_lv`, сток). Половина функциональности (поиск, заказы, отзывы, избранное залогиненных) написана против БД и не работает с хардкодом; вторая половина (витрина, sitemap) — против хардкода и не масштабируется (изменение цены = редеплой). Это корневая причина большинства багов из отчёта 02.

## Решение

Каталог (товары, категории, бренды, таксономия) живёт **только** в Supabase/Postgres. `lib/data.ts` конвертируется в `supabase/seed.sql` и удаляется. Идентификатор товара — UUID (строка) во всём приложении: корзина, избранное, отзывы, аналитика. Витрина читает каталог через `lib/commerce` (ADR-0004) с кэшированием ISR/tags (ADR-0006).

## Последствия

- Проще: одна модель, работающие поиск/заказы/избранное, управление каталогом без деплоя, путь к админке и 100k SKU.
- Сложнее: витрина зависит от доступности БД → обязательна кэш-стратегия (ISR); нужен seed и миграция потребителей (шаг B6 blueprint).
- Необратимо-дорогое: смена ID на UUID ломает старые localStorage-корзины/избранное — допустимо до запуска (версионирование ключа).
- Затронут Playbook §15.3.

## Альтернативы

- **Оставить data.ts как источник, БД только для заказов** — отклонено: не масштабируется, консервирует двойную модель, блокирует админку и поиск.
- **Headless CMS (Sanity/Payload) для каталога** — отклонено сейчас: третья система при команде 1–2 человека; БД уже спроектирована; CMS возможна позже поверх той же БД (PIM, Phase 4).
