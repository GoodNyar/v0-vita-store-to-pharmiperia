# ADR-0005: Вебхук Stripe — единственный источник факта оплаты; жизненный цикл заказа на сервере

- **Статус:** принят
- **Дата:** 2026-07-03
- **Авторы:** Engineering (по итогам аудитов, отчёты 02, 04)

## Контекст

Исторически заказ создавался клиентом в колбэке `onComplete` со статусом `'paid'`, цены для Stripe-сессии приходили из браузера, вебхука не было. Следствия: подмена цены возможна, «оплаченный» заказ создаётся без оплаты (или не создаётся при закрытой вкладке), сверка платежей с заказами невозможна, письма/сток/аналитику нельзя привязать к достоверному событию.

## Решение

1. Server action создания сессии принимает только `{productId, quantity}[]`; цены читает из БД; создаёт draft-заказ и кладёт `order_id` в metadata сессии.
2. `app/api/webhooks/stripe` (проверка подписи, идемпотентность по `event.id`) — единственное место, переводящее заказ в `paid` по `checkout.session.completed`.
3. Статусы заказа — конечный автомат на сервере (`draft → pending → paid → fulfilled/refunded/cancelled`); клиент не инициирует переходы.
4. UI-колбэки Stripe используются только для навигации на страницу «спасибо», которая читает статус с сервера.
5. Побочные эффекты (`письмо`, `сток`, далее лояльность/аналитика) вызываются из вебхука; при появлении третьего эффекта — вынос в событийный каркас (blueprint E3).

## Последствия

- Проще: платёж достоверен, reconciliation со Stripe тривиален, гостевые заказы возможны (insert через server-side с service-role), возвраты получают точку опоры.
- Сложнее: нужен staging со Stripe CLI для локальной отладки вебхуков; «спасибо-страница» должна уметь состояние «оплата подтверждается» (вебхук асинхронен).
- Затронут Playbook §1.1, §1.5, §15.2, §15.10.

## Альтернативы

- **Доверять `onComplete` + сверять позже** — отклонено: окно недостоверности + клиентский вызов подделывается.
- **Синхронный опрос сессии после редиректа вместо вебхука** — отклонено как единственный механизм: теряется при закрытой вкладке; допустим как ускоритель UX поверх вебхука.
- **Payment Intents + собственная форма** — отклонено: расширяет PCI-scope и объём кода; Embedded Checkout сохраняем (отчёт 03 §8).
