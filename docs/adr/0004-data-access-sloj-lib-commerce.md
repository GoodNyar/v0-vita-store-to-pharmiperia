# ADR-0004: Data-access-слой `lib/commerce`; запрет прямых запросов из компонентов

- **Статус:** принят
- **Дата:** 2026-07-03
- **Авторы:** Engineering (по итогам аудитов, отчёты 02–03)

## Контекст

Supabase-клиент вызывается напрямую из ~15 страниц и компонентов. Следствия: схема БД является публичным API всего приложения (любое изменение схемы — breaking change для всех экранов), рукописные интерфейсы строк таблиц дрейфуют от реальной схемы (причина неработающих поиска/заказов/отзывов), бизнес-логика непереиспользуема (мобильное приложение потребует второй реализации), невозможно централизованно кэшировать и логировать.

## Решение

Весь доступ к данным — через модуль `lib/commerce/` (products, cart, orders, favorites, reviews, search, admin). Правила:
1. `createClient()` (browser/server) импортируется только внутри `lib/` — enforced через ESLint `no-restricted-imports`.
2. Функции слоя принимают/возвращают доменные типы, построенные на сгенерированных типах БД (`supabase gen types` в CI).
3. Компоненты и страницы не знают имён таблиц и колонок.
4. Ошибки слоя типизированы; «проглотить и вернуть пустое» запрещено.

Миграция — strangler (blueprint, шаг B3): слой создаётся рядом, потребители переводятся по одному, lint-правило включается последним.

## Последствия

- Проще: изменение схемы локализовано; будущий Storefront API (Phase 5) и админка используют тот же слой; кэш/логирование в одной точке.
- Сложнее: +1 уровень косвенности; дисциплина «не ходить в БД напрямую» требует lint-enforcement, не договорённости.
- Затронут Playbook §1.3, §15.4.

## Альтернативы

- **Оставить прямые запросы, полагаясь на RLS** — отклонено: RLS решает авторизацию, но не контракт, не дрейф типов и не переиспользование.
- **Сразу tRPC/GraphQL-прослойка** — отклонено: лишний рантайм-слой для команды 1–2 человека; `lib/commerce` даёт ту же границу без транспорта; транспорт добавится при появлении мобильного клиента (ADR потребуется).
- **Repository-классы/ORM (Prisma/Drizzle)** — отклонено сейчас: Supabase-клиент уже даёт типизированный доступ + RLS; смена ORM — отдельное решение со своим ADR при реальной боли.
