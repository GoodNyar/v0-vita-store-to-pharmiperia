# ADR-0003: URL-схема — слаги товаров и локаль как сегмент пути

- **Статус:** принят
- **Дата:** 2026-07-03
- **Авторы:** Engineering (по итогам аудитов, отчёты 03–04)

## Контекст

Текущие URL: `/products/1` (числовой id), язык хранится в localStorage и не существует в URL — контент по одному адресу меняется в зависимости от состояния клиента. Следствия: для Google существует только один язык (hreflang нечем выразить), числовые URL проигрывают в SEO, смена URL после индексации = потери позиций через редиректы. URL — контракт с поисковиком на годы; это самое необратимое решение проекта.

## Решение

1. Товары и разделы адресуются слагами: `/products/bioderma-sensibio-h2o-500ml`, `/category/skincare`, `/brand/avene`.
2. Локаль — первый сегмент пути: `/lv/...`, `/ru/...`; `<html lang>` и словари определяются из URL. Корень `/` — 301/307 на локаль по `Accept-Language`/cookie.
3. Слаг-генерация — одна функция в `lib/commerce/slugs.ts` (нормализация + транслитерация диакритики: `Avène → avene`).
4. Опубликованный URL меняется только через 301 с записью в реестре редиректов.

## Последствия

- Проще: двуязычная индексация, hreflang, человекочитаемые URL, SEO-масштабирование на новые страны (Phase 5) — добавлением локали.
- Сложнее: перенос дерева `app/` под `[locale]` (шаг C1 blueprint); middleware-редиректы со старой схемы; slug — новая уникальная колонка, обязательная при создании товара.
- Необратимо: после индексации схема фиксируется навсегда.
- Затронут Playbook §1.8, §15.6.

## Альтернативы

- **Локаль в поддомене (`ru.pharmiperia.lv`)** — отклонено: дробит SEO-авторитет домена, усложняет cookie/auth; поддиректории — рекомендация Google для языковых версий одного рынка.
- **Локаль в query/cookie (как сейчас)** — отклонено: невидимо для краулера, блокирует SEO второго языка.
- **`?id=` рядом со слагом** — отклонено: слаг уникален в БД, id в URL не нужен.
