# Отчёт 03 — Архитектурный аудит масштабируемости

> Дата: 2026-07-02 · Перспектива: Staff Engineer (Vercel/Shopify/Stripe) · Горизонт: 100k SKU, десятки тысяч пользователей, мультистраны/языки/валюты, мобильное приложение, админка

**Главный тезис:** Pharmiperia сегодня — это не магазин с фронтендом, а **фронтенд без магазина**. Нет коммерческого ядра как слоя: нет доменной модели (Product, Cart, Order, Price с инвариантами), нет API-границы, нет событийной модели. Вся «логика» размазана по клиентским компонентам. Каждое требование масштаба упирается в это.

## 1. Что покажется инженерам непрофессиональным

- **Каталог как TS-массив в бандле** — изменение цены = редеплой.
- **Прямые запросы из браузера в БД по всему приложению** — схема БД является публичным API; отсутствие data-access-слоя.
- **Перевёрнутая модель рендеринга** — статичный SEO-контент рендерится на клиенте, а auth-refresh в middleware выполняется на каждый запрос каждой страницы.
- **Заказ создаётся клиентом со статусом `paid`** — жизненный цикл заказа управляется браузером; без идемпотентности, вебхука, серверной state machine.
- **Состояние «кто я» размазано** — четыре независимых источника истины об одной сессии.
- **`ignoreBuildErrors: true`** — решение отключить единственный механизм контроля качества.
- **Числовые ID в публичных URL** — URL это контракт с поисковиком на годы.
- **Метаданные заказа как JSON.stringify в Stripe metadata** — лимит 500 символов, ломается на ~8 позициях корзины.

## 2. Что выдаёт AI-генерацию

- `console.log("[v0] ...")` в cart-context и account.
- README от v0, `"name": "my-project"`.
- Дословные комментарии из документации Supabase в proxy.ts.
- Четыре копии `getBrandSlug` с расходящимися реализациями.
- Два несовместимых словаря корзины (`addToCart` / `addItem`).
- 24 страницы, оборачивающие себя в провайдеры повторно.
- 57 shadcn-компонентов при используемых ~5; неиспользуемые recharts/embla/vaul/input-otp.
- Комментарии-алиби: «would come from CMS in real app», «prices validated server-side».
- Мёртвые компоненты целиком; дублирующиеся SQL-скрипты.
- Смесь языков в хардкодах (латышский/русский/английский по разным сессиям генерации).

## 3. Что переписать, чтобы происхождение было неопределимо

Происхождение выдаёт не стиль, а отсутствие следов принятия решений:
- **Слой данных**: единый `lib/commerce/` с репозиториями и типами из `supabase gen types`; запрет `createClient()` в компонентах.
- **Композиция страниц**: route groups, layout-композиция сверху вниз; разбить монолиты (account 1319 строк).
- **i18n**: словари по namespace + locale-роутинг вместо рукописного контекста на 1787 строк с `t(key as any)`.
- **Гигиена**: удалить мёртвый код, дубли SQL, дефолтный README, `[v0]`-логи, лишние зависимости; git с осмысленной историей.

## 4. Что ограничит масштабирование (по параметрам)

**100 000 товаров:** каталог в бандле умирает на ~10³ SKU; в схеме нет вариантов (product → variants) и атрибутов/метаполей; поиск ILIKE без pg_trgm/tsvector и фасетов; изображения в /public с unoptimized; SSG всех товаров на каждый деплой (нужна ISR/tags).

**Нагрузка/пользователи:** middleware дергает Supabase Auth на каждый HTML-запрос; ни одного слоя кэширования; AI-эндпоинты шлют 50 товаров в промпт каждому анониму.

**SEO-конкуренция:** CSR-каталог (Google видит пустой div); язык не существует в URL → второго языка для поисковика нет, hreflang нечем выразить; числовые URL без слагов; canonical всех страниц указывает на корень.

**Несколько стран:** «Латвия» зашита повсюду (пакоматы, доставка, НДС отсутствует). Нет сущности **Market** (страна → валюта, язык, доставка, налоги).

**Несколько языков:** `Lang = "ru" | "lv"` в типе и `name_ru/name_lv` в колонках — третий язык = миграция схемы. Нужны таблицы переводов. Все переводы UI — одним куском в бандл.

**Несколько валют:** цена — float EUR в десятке мест; нет Money-типа, минорных единиц, прайс-листов.

**Мобильное приложение:** API нет; Server Actions — невесионированный RPC, из нативного кода недоступен; логика в React-компонентах → приложение = вторая реализация всей бизнес-логики.

**Админка:** в RLS нет ни одной админской политики — не существует роли, способной создать товар; нет RBAC, audit log, soft-delete.

## 5. Что менять сейчас, пока проект молодой

1. **URL-схема**: слаги + локаль в пути — после индексации миграция с потерями.
2. **Деньги как минорные единицы + код валюты** — пока ~30 точек использования.
3. **Модель каталога в БД**: variants, таблицы переводов, атрибуты.
4. **Единый data-access-слой**, запрет прямых запросов из компонентов.
5. **Серверный жизненный цикл заказа** (webhook-driven state machine).
6. **Сущность Market** вместо латвийских констант в JSX.
7. **Инженерная база**: git, CI с typecheck, генерация типов из схемы.

## 6. Что потребует полного переписывания через 2–3 года, если не трогать

- Чекаут и заказ (клиентская оркестровка не расширяется до возвратов/инвойсов).
- i18n-система (не доживёт до третьего языка).
- Модель данных каталога (миграция под нагрузкой).
- Корзина (снапшоты товаров в localStorage).
- CSR-оболочка витрины.
- AI-фичи («каталог в промпте», открытые endpoints).

## 7. Отсутствующие практики крупных e-commerce платформ

Разделение commerce-ядра и витрины · событийная обработка заказа · идемпотентность платежей · inventory reservations · поисковый сервис (фасеты, синонимы, морфология LV/RU) · PIM/CMS · промо-движок · ISR/cache tags · observability · feature flags и A/B · data pipeline (warehouse) · consent management · контрактные типы и версионирование API.

## 8. Что удачно и не менять

- Выбор стека (Next + Supabase + Stripe + Tailwind/shadcn).
- SQL-схема как черновик ядра (RLS, триггеры, JSONB-адреса, снапшот цены в order_items).
- SEO-обвязка серверных layout'ов (metadata, JSON-LD, sitemap/robots).
- Дизайн-система на CSS-переменных (oklch, темы).
- Guest-first UX (корзина/избранное без логина + merge).
- `@supabase/ssr`-паттерн (сузить, не заменять).
- Stripe Embedded Checkout (минимальный PCI-scope).

## 9. Замечания технического ревью (по убыванию)

1. *Blocking:* витрина не имеет серверной поверхности данных — data-access-слой и модель кэширования до любого фичекода.
2. *Blocking:* жизненный цикл заказа управляется клиентом — вебхук, state machine, идемпотентность.
3. *Blocking:* локаль не в URL — структура URL до запуска, потом невозвратно.
4. *Major:* деньги — float без валюты; Money-тип до второго прайса.
5. *Major:* нет вариантов и атрибутов; `name_ru/name_lv` не масштабируется.
6. *Major:* рендеринг-модель инвертирована; определить static/dynamic/ISR по маршрутам.
7. *Major:* нет роли admin — система не управляема после запуска.
8. *Major:* AI-эндпоинты без авторизации, бюджета, ретривала.
9. *Minor:* 4 slug-функции, двойной словарь корзины, дубли провайдеров.
10. *Minor:* нет git/CI/typecheck-гейта; зависимости не соответствуют использованию.
11. *Process:* зафиксировать выход из статуса прототипа: ADR на ключевые решения, чистка артефактов, code ownership.

## Итог

Стек правильный, схема БД — хороший черновик, UX-скелет удачен. Но проект — витрина-прототип без коммерческого ядра. Четыре решения принять сейчас (необратимы потом): **URL (слаги + локаль), деньги в минорных единицах, модель каталога (варианты + переводы), API-граница**. Всё остальное ложится на эти опоры; без них каждое требование масштаба оплачивается переписыванием.
