# Отчёт 06 — Refactoring Blueprint: технический roadmap на год

> Дата: 2026-07-03 · Роль: Lead Staff Engineer · Команда: 1–2 разработчика
> Опирается на отчёты 02–05 и Playbook; находки не повторяются — используются как принятые решения (см. `docs/adr/`).
> Принцип: **strangler, не rewrite** — каждый шаг оставляет проект рабочим и деплоибельным.

## Часть 1. Логические модули проекта

Целевая модульная карта (модуль ≠ папка; папки — ниже):

| Модуль | Что содержит | Где живёт сейчас | Где должен жить |
|---|---|---|---|
| **Commerce Core** | товары, корзина, заказы, цены, промо | размазан: `lib/data.ts`, провайдеры в `components/`, inline-запросы в страницах | `lib/commerce/` (создать) |
| **Payments** | Stripe-сессии, вебхук, Money | `app/actions/stripe.ts`, `lib/stripe.ts` | `lib/stripe/` + `lib/money.ts` + `app/api/webhooks/stripe/` |
| **Identity & Access** | auth-клиенты, middleware, RLS, роли | `lib/supabase/`, `components/auth-provider.tsx` | там же, суженный |
| **Storefront UI** | страницы витрины, feature-компоненты | `app/*`, `components/*` (плоско) | `app/[locale]/(shop|content|account)/`, `components/<feature>/` |
| **Localization** | словари, локаль-роутинг, форматирование | `lib/i18n.tsx` (монолит 1787 строк) | `lib/i18n/` (namespace-словари) |
| **Content** | блог, промо-тексты, описания товаров | JSX страниц + словарь i18n | БД (переводы контента) + `(content)`-группа |
| **AI Services** | чат, рекомендации | `app/api/chat`, `app/api/recommendations` | там же, за общим guard-слоем |
| **Data Platform** | схема, миграции, типы | `scripts/*.sql` (дрейфует) | `supabase/migrations/` + генерируемые типы |
| **Design System** | токены, ui-примитивы | `app/globals.css`, `components/ui/` (57 файлов) | те же, вычищенные |
| **Ops/Observability** | ошибки, логи, CI | отсутствует | CI + Sentry + error boundaries |

Корневая причина всех проблем ниже: **у Commerce Core нет дома** — поэтому каждая страница реализует его кусочек заново.

## Часть 2. Анализ по папкам

### 2.1 `app/` (40 страниц)

- **Текущая ответственность:** маршрутизация, композиция страниц и — ошибочно — вся бизнес-логика (fetch в `useEffect`, расчёты, правила доставки).
- **Архитектурные проблемы:** страницы — самодостаточные монолиты; route groups не используются; layout'ы есть только у 3 поддеревьев из ~10 — остальные страницы сами рендерят шапку/футер/провайдеры; клиентский рендеринг по умолчанию.
- **Технический долг:** 24 страницы с дублирующими обёртками `LangProvider`/`CartProvider`; собственные локальные копии `Product`-интерфейса в `category/[slug]` и `search` (расходятся с `lib/data.ts` и со схемой БД); собственная копия `getBrandSlug` в `category`; латвийские константы доставки в `checkout/page.tsx`.
- **Code smells:** файлы 500–1300 строк (`account/page.tsx` — 1319); `t(key as any)`; `window.location.href` для навигации; копипаст между `popular` и `specials` (одна страница с параметром); формы-имитации (`contact`).
- **Риски масштабирования:** каждая новая страница линейно наращивает дублирование; изменение общего каркаса = правка десятков файлов.
- **Связанность:** страницы связаны одновременно со схемой Supabase (напрямую), с форматом localStorage, с ключами словаря i18n и друг с другом через копипаст.
- **Поддерживаемость:** низкая; локализация изменения невозможна — «поменять шапку» затрагивает все автономные страницы.

### 2.2 `app/api/` + `app/actions/`

- **Ответственность:** AI-чат, AI-рекомендации, создание Stripe-сессии.
- **Проблемы:** нет слоя валидации входа (zod отсутствует при том, что установлен); нет rate-limit; бизнес-логика inline в хендлерах; нет общего формата ошибок/ответов.
- **Долг:** `recommendations` написан под несуществующую схему; `actions/stripe.ts` принимает цены с клиента.
- **Связанность:** прямое сцепление хендлеров со схемой БД и конкретной моделью (`openai/gpt-4o-mini` строкой).
- **Риск:** это единственная серверная поверхность проекта — и она без контрактов; мобильное приложение (Phase 5 roadmap) не сможет на неё опереться.

### 2.3 `components/` (26 feature-компонентов)

- **Ответственность:** де-факто здесь живёт слой состояния приложения (4 провайдера) + шапка/футер/карточка/корзина — «components» содержит домен, что само по себе smell.
- **Проблемы:** провайдеры (`cart-context`, `favorites-provider`) — это Commerce Core, замаскированный под UI; `product-card` смешивает навигацию, избранное, корзину и i18n; плоская структура без feature-папок.
- **Долг:** мёртвые `quick-view-modal`, `pull-to-refresh`, `theme-provider`, `category-sidebar`; `site-header` (507 строк) содержит копию slug-функции и три меню в одном файле; два тост-механизма (`hooks/use-toast`+`toaster` и `sonner` в зависимостях).
- **Smells:** `console.log("[v0]")`; самодельные оверлеи (корзина, сайдбар) вместо Radix; бизнес-правила в JSX (`favorites`-merge в провайдере).
- **Связанность:** компоненты импортируют `lib/data` (сцепка с хардкод-каталогом), Supabase-клиент (сцепка со схемой), строковые ключи словаря.
- **Поддерживаемость:** средняя — код читаемый, но границы отсутствуют; перенос корзины на сервер потребует трогать и провайдер, и каждый компонент-потребитель.

### 2.4 `components/ui/` (57 файлов)

- **Ответственность:** shadcn-примитивы.
- **Долг:** ~50 не используются; риск дрейфа от upstream при точечных правках.
- **Риск:** низкий по рантайму (tree-shaking), высокий по шуму: новые разработчики теряются, велик соблазн добавить дубликат.
- **Действие:** сократить до используемых; добавлять по мере надобности генератором (правило Playbook §3).

### 2.5 `lib/`

- **`data.ts`** — три ответственности в одном файле: каталог (данные), таксономия (конфигурация), brand-хелперы (логика). Комментарий «single source of truth for brand slugs» уже опровергнут тремя копиями функции по проекту. Судьба: каталог → seed БД; таксономия → БД; хелперы → `lib/commerce/brands.ts`.
- **`i18n.tsx`** — монолит: словарь RU/LV + провайдер + `formatEur` + переводы описаний товаров (`productDescriptions` по id — контент товаров в UI-словаре, прямой smell). Клиентский, целиком в бандле, `t()` без плюрализации. Судьба: `lib/i18n/` с namespace-словарями, серверная загрузка, контент — в БД.
- **`stripe.ts`, `utils.ts`** — корректны, остаются.
- **`supabase/`** — канонический паттерн, но: `client.ts` парсит cookie вручную (`split("=")` теряет значения с `=`); `proxy.ts` неверно назван (это middleware-session). Судьба: заменить самодельный cookie-код дефолтом `createBrowserClient`, переименовать в `middleware.ts`-хелпер.
- **Отсутствует:** `money.ts`, `commerce/`, `events/` — их создание и есть ядро blueprint.

### 2.6 `hooks/`

- `use-mobile` — ок. `use-toast` — сирота, используется только `ui/toaster.tsx`, при живом `sonner` в зависимостях. Выбрать одну систему тостов (рекомендация: sonner), вторую удалить.

### 2.7 `scripts/`

- **Ответственность:** SQL-схема — фактически единственная документация модели данных.
- **Долг:** `001_create_schema.sql` дублирует `001_create_tables.sql`+`002_rls_triggers.sql`; `003` ≈ `004`; нет связи с CLI-конвейером; схема дрейфует от кода (loyalty-таблиц нет, код их читает).
- **Судьба:** конвертировать в `supabase/migrations/` (одна консолидированная baseline-миграция + seed), папку `scripts/` удалить.

### 2.8 `public/`

- **Ответственность:** статика. Логотипы брендов/платёжек/доставки — на месте; товарные изображения и мета-ассеты (og, иконки) — отсутствуют при наличии ссылок.
- **Риск масштабирования:** «изображения в git» умирает при живом каталоге; целевое место — Supabase Storage/CDN (Phase 4 roadmap), сейчас — просто восполнить недостающее.

### 2.9 `styles/`, корень

- `styles/globals.css` — мёртвый дубликат (используется `app/globals.css`). Удалить.
- Корень: `popular.png`, `specials.png`, `tsconfig.tsbuildinfo` — убрать/в .gitignore; `package.json name: "my-project"` → `pharmiperia`; `.env.example` создать.

## Часть 3. План исполнения

Формат: **приоритет** (P0 — фундамент/блокирует остальное, P1 — ядро года, P2 — по триггеру) · **оценка** в человеко-днях (pd) для разработчика, знакомого со стеком · **зависимости** · **эффект** · **файлы** · **стратегия миграции**.

Правило параллелизации для 2 разработчиков: дорожка «Платформа» (A→B→C) и дорожка «UI-гигиена» (может идти параллельно с B).

### Этап A — Фундамент (недели 1–2)

**A1. Git + CI + возврат typecheck — P0, 2–3 pd, зависимостей нет**
- Эффект: все последующие шаги становятся безопасными; компилятор бесплатно находит фантомные контракты.
- Файлы: `next.config.mjs` (убрать `ignoreBuildErrors`), `package.json` (eslint + конфиг), `.github/workflows/ci.yml`, `.env.example`; починка всплывших TS-ошибок (`clearCart`, `addItem`, товары 13–16 в `lib/data.ts`).
- Миграция: сначала `git init` + коммит as-is (точка отката), затем включение проверок; TS-ошибки чинятся в том же PR, где включён typecheck — не «потом».

**A2. Чистка репозитория — P0, 1–2 pd, зависит от A1**
- Эффект: −5 мёртвых компонентов, −50 ui-файлов, −6+ зависимостей, один globals.css, один toast; исчезают «AI-артефакты» (отчёт 03 §2).
- Файлы: `components/{quick-view-modal,pull-to-refresh,theme-provider,category-sidebar}.tsx`, `components/ui/*` (неиспользуемые), `styles/`, `hooks/use-toast.ts`, `package.json`, корневые PNG, README.
- Миграция: удаление по данным `grep`-подтверждения неиспользования; один PR, чтобы откат был атомарным.

**A3. Границы ошибок + Sentry — P0, 2 pd, зависит от A1**
- Эффект: белый экран невозможен; сбои видимы команде, а не покупателю.
- Файлы: `app/global-error.tsx`, `app/not-found.tsx`, `app/[locale]`-варианты позже; `error.tsx`/`loading.tsx` для сегментов с данными; `instrumentation.ts` (Sentry), удаление `[v0]`-логов.
- Миграция: аддитивно, риска нет.

### Этап B — Извлечение ядра (недели 3–6)

**B1. Миграции через Supabase CLI + staging — P0, 2 pd, зависит от A1**
- Эффект: схема перестаёт дрейфовать; появляется безопасная площадка для B4–C2.
- Файлы: `supabase/migrations/0001_baseline.sql` (консолидация `scripts/*` + недостающие loyalty-таблицы или их осознанное исключение), `supabase/seed.sql`; удаление `scripts/`.
- Миграция: baseline фиксирует *текущее* прод-состояние (`db pull`), дальше — только новые миграции. Staging-проект пересоздаётся из миграций как проверка.

**B2. Генерация типов БД в CI — P0, 1 pd, зависит от B1**
- Эффект: контракт БД↔код перестаёт дрейфовать; рукописные интерфейсы в страницах получают замену.
- Файлы: `lib/database.types.ts` (генерируемый), CI-шаг сверки.
- Миграция: типы добавляются, старые интерфейсы умирают по мере шагов B3/B5.

**B3. `lib/commerce/` — скелет + пилот (strangler) — P0, 3–4 pd, зависит от B2**
- Эффект: появляется дом Commerce Core; каждая следующая фича пишется уже правильно.
- Файлы: `lib/commerce/{products,favorites,orders,reviews}.ts`; пилотные потребители — `favorites-provider.tsx` и `app/account/orders/page.tsx` (два самых изолированных места с прямыми запросами).
- Миграция: strangler — слой создаётся рядом, call-sites переводятся по одному; ESLint-правило `no-restricted-imports` на `@/lib/supabase/client` вне `lib/` включается, когда переведён последний потребитель (маркер завершения).

**B4. `lib/money.ts` + центы — P1, 2–3 pd, зависит от A1 (лучше после B1) — ADR-0002**
- Эффект: мультивалютная готовность; исчезает float-арифметика на деньгах.
- Файлы: `lib/money.ts` (новый), `lib/data.ts`/seed (цены → центы), `cart-context`, `checkout/page.tsx`, `actions/stripe.ts`, `formatEur` → `formatMoney`; миграция колонок БД `DECIMAL`→`INTEGER` в новой миграции.
- Миграция: тип + конверсия на границах в один PR; версия ключа localStorage корзины (`pharmiperia:v1:cart` → `v2`) со сбросом старого формата — корзины гостей потерять допустимо до запуска.

**B5. Консолидация корзины и провайдеров — P0, 3 pd, зависит от A1**
- Эффект: один словарь (`addCartLine`, `removeCartLine`, `setLineQuantity`, `clearCart`), одно дерево провайдеров; −24 дублирующих обёртки; чинится «В корзину» на поиске/категориях и пост-оплатный краш.
- Файлы: `components/cart-context.tsx`, все 24 страницы с обёртками, `app/search/page.tsx`, `app/category/[slug]/page.tsx`, `app/checkout/*`.
- Миграция: сначала расширить контекст недостающими методами (аддитивно), затем механическая замена вызовов, затем снятие обёрток; каждая стадия — отдельный зелёный коммит.

**B6. Каталог в БД, отставка `lib/data.ts` — P0, 4–5 pd, зависит от B1–B3 — ADR-0001**
- Эффект: один источник истины; поиск/рекомендации/избранное/заказы начинают работать против реальных данных; UUID-строковый ID везде.
- Файлы: `supabase/seed.sql` (16 товаров + категории + бренды из data.ts), `lib/commerce/products.ts` (queries с `is_active`, `name_ru/name_lv`), все страницы-потребители `products from "@/lib/data"`, `sitemap.ts` (из БД), `products/[id]/layout.tsx`.
- Миграция: seed → `lib/commerce` читает БД → страницы переводятся по одной (данные идентичны, регрессия видна глазами) → `data.ts` сокращается до таксономии, затем таксономия тоже в БД → файл удаляется. Изображения товаров добавляются в этом же шаге.

### Этап C — Необратимые решения (недели 7–10)

**C1. URL: слаги + `[locale]`-сегмент + 301 — P0, 4–5 pd, зависит от B6 — ADR-0003**
- Эффект: SEO-контракт зафиксирован навсегда; двуязычная индексация становится возможной.
- Файлы: `app/[locale]/` (перенос дерева), `middleware.ts` (locale-детект + 301 со старых `/products/1` на `/lv/products/<slug>`), `lib/commerce/slugs.ts` (единственная slug-функция), `sitemap.ts`, `robots.ts`, все `<Link>`-потребители.
- Миграция: перенос дерева — механический (git mv); реестр редиректов — таблица/константа со старыми URL; неделю после — мониторинг 404 в Sentry.

**C2. Серверный путь заказа — P0, 4–5 pd, зависит от B3, B4 — ADR-0005**
- Эффект: оплата и заказ становятся достоверными; открывается точка для писем/аналитики/стока.
- Файлы: `app/api/webhooks/stripe/route.ts` (новый), `app/actions/stripe.ts` (id+qty вместо цен; server-side lookup цен; metadata = order_id), `lib/commerce/orders.ts` (создание draft-заказа до сессии, подтверждение из вебхука), `checkout/page.tsx` (убрать клиентскую запись), миграция RLS для гостевых заказов (insert через service-role в action).
- Миграция: параллельный запуск — вебхук пишет заказы, старый клиентский путь удаляется только после первого подтверждённого тестового платежа на staging; идемпотентность по `event.id`.

**C3. i18n-реструктуризация — P1, 4–5 pd, зависит от C1 — ADR-0007**
- Эффект: словари по namespace, серверная загрузка, `<html lang>` из URL; −1787-строчный монолит из клиентского бандла; контент товаров уходит в БД (у B6 уже есть `description_ru/lv`).
- Файлы: `lib/i18n/{config,dictionaries/*}.ts`, `lib/i18n.tsx` (умирает), все потребители `useLang` (сигнатура `t()` сохраняется — миграция дешёвая), `app/[locale]/layout.tsx`.
- Миграция: словарь режется скриптом по префиксам ключей; провайдер сохраняет API на переходный период; хардкод-строки (checkout, AI-квиз) добираются в словари в этом же шаге.

### Этап D — Рендеринг и декомпозиция (месяцы 3–5)

**D1. RSC/ISR витрины постранично — P1, 8–10 pd, зависит от B6+C1+C3 — ADR-0006**
- Эффект: SEO-контент в HTML; TTFB/LCP из кэша; middleware-auth уходит со статики.
- Файлы: `app/[locale]/(shop)/{page,products,category,brand,popular,specials}.tsx`, `(content)/*`; клиентские островки (`AddToCartButton`, `FavoriteButton`) выделяются из `product-card`.
- Миграция: по одной странице в PR, начиная с контентных (нулевой риск) → категории → товар → главная; каждая объявляет `revalidate`/tags; сравнение snapshot-рендера до/после.

**D2. Декомпозиция `account/page.tsx` — P2, 3–4 pd, независим**
- Эффект: 1319 строк → подмаршруты `account/{profile,notifications}` + существующие; удаление PII-кэша из localStorage и самодельного тоста.
- Миграция: вырезать секции в файлы без изменения логики, затем чистить.

**D3. Поиск v1 — P1, 3–4 pd, зависит от B6**
- Эффект: поиск начинает находить; параметризованные фильтры закрывают PostgREST-инъекцию.
- Файлы: миграция `pg_trgm` + `tsvector`-колонка + индекс, `lib/commerce/search.ts`, `app/[locale]/(shop)/search/page.tsx` (RSC + searchParams).
- Миграция: аддитивная; страница переводится вместе с D1-волной.

**D4. RBAC + `/admin` v0 — P1, 4–5 pd, зависит от B1, B3**
- Эффект: операции (заказы, сток) без Supabase Studio; admin-политики RLS.
- Файлы: миграция (роль в JWT-claims + политики), `app/admin/{layout,orders,products}/`, `lib/commerce/admin.ts`.
- Миграция: аддитивно; доступ по allow-list user_id до полноценных ролей.

### Этап E — Закрепление (фоново с этапа C)

**E1. Тесты денежного пути — P0 (для C2), 3–4 pd**
- unit: `lib/money`, cart-редьюсер, слаги; e2e: checkout happy-path на staging (Stripe test) — включается в CI сразу после C2.
- Файлы: `tests/unit/*`, `tests/e2e/checkout.spec.ts`, CI.

**E2. AI-поверхность — P1 (guard) / P2 (retrieval), 2–3 pd + позже**
- Guard сейчас: zod-валидация, rate limit (Upstash), лимит длины, бюджет-алерты — `app/api/{chat,recommendations}/route.ts`. Retrieval (pgvector) — по триггеру роста каталога.

**E3. Событийный каркас — P2, 4–5 pd, зависит от C2, триггер — вторая «реакция на заказ»**
- Пока реакций на `order.paid` одна-две (письмо, сток) — достаточно прямых вызовов из вебхука. Каркас (Inngest/Queues) вводится, когда появляется третья (лояльность/аналитика), не раньше — избегаем преждевременной инфраструктуры.
- Файлы: `lib/events/`, рефактор вебхука на publish.
