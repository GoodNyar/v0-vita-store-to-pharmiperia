# Top High Issues

### H-1 · Веб-checkout не привязывает разрешённый рынок → все заказы жёстко `market_code='lv'` + НДС 21%
**Где:** `app/actions/stripe.ts:42` — `createCheckoutSession` вызывает `createDraftOrder(items, { ...customer, userId })`, не считывая cookie `pharm_market` (нет `cookies()` в экшене). `app/[locale]/checkout/page.tsx` и `components/stripe-checkout.tsx` нигде не задают `customer.marketCode` (grep пуст). Следовательно `createDraftOrder` получает `marketCode: undefined` → `DEFAULT_MARKET_CODE` (`lib/orders.ts:142`) = `'lv'`.
**Почему это проблема:** middleware разрешает рынок (`resolveMarket`: header → cookie → geo → default) и выставляет cookie `pharm_market` (`middleware.ts:66-76`), Storefront API принимает `?market=` и учитывает его — но путь создания веб-заказа этот рынок полностью игнорирует. Все headline-возможности фазы (market VAT, OSS-атрибуция) в реальном заказе не задействованы.
**Насколько критично:** High. При LV-only — безвредно (всё и так `lv`). При запуске LT/EE — активируется как дефект: заказы из Эстонии будут облагаться 21% вместо 22% (`markets.ee.vat_rate_bps = 2200`) и атрибутироваться как `lv` → OSS-отчётность неверна; `validateShippingForMarket` проверяет доставку против рынка `lv`, а не фактического.
**Последствия:** налоговая некорректность и неверная OSS-атрибуция ровно для того сценария (мульти-рынок), ради которого Phase 6 существует.
**Рекомендация:** читать `pharm_market` через `cookies()` внутри `createCheckoutSession` и передавать в `createDraftOrder` (или разрешать рынок на сервере из заголовков/cookie и не доверять клиентскому вводу). До этого не считать «international checkout» поставленным.

# Top Medium Issues

### M-1 · Три расходящихся пути ценообразования; веб-витрина игнорирует market pricing
**Где:** `lib/commerce/catalog-source.ts` не содержит market pricing (grep `market` пуст) → веб-каталог/PDP/корзина показывают каталожную цену. При этом `lib/orders.ts:97` и `lib/commerce/storefront.ts:78` применяют `getMarketPriceForProduct`.
**Почему это проблема:** сегодня таблица `market_product_prices` без seed → override всегда null → `resolveMarketPrice` возвращает каталожную цену, все три пути совпадают. Но как только админ добавит override: веб-UI покажет каталожную цену, checkout спишет market-цену (для рынка `lv` из-за H-1), Storefront API вернёт market-цену запрошенного рынка → расхождение «показано ≠ списано» и «web ≠ API».
**Критичность:** Medium (латентно, нет seed). **Рекомендация:** либо провести market pricing в веб-витрину, либо явно задокументировать, что override-ы действуют только через API до подключения витрины.

### M-2 · Резервации выключены флагом, но неполный жизненный цикл за флагом не изменён
**Где:** `lib/inventory/reserve.ts` — `reserveInventoryForOrder`/`releaseInventoryReservation` возвращают рано, если `INVENTORY_RESERVATIONS_ENABLED !== 'true'`; `lib/orders.ts` больше не вызывает reserve (grep пуст).
**Почему это проблема:** это корректно нейтрализует побочный эффект Phase 5 (занижение доступности + рост таблицы) — сегодня резервации не создаются. Но реализация за флагом не завершена: нет consume/release на `order.paid`, нет cron истечения, checkout не гейтится. Включение флага в prod вернёт дефект Phase 5 H-1 в полном объёме.
**Критичность:** Medium (dormant). **Рекомендация:** держать выключенным; завершить жизненный цикл до включения; пометить флаг небезопасным в конфиге/доке.

### M-3 · Per-market shipping в БД, но веб-checkout использует хардкод-опции
**Где:** self-audit признаёт «Checkout UI hardcoded options; server validates cost». `validateShippingForMarket(cost, market)` проверяет стоимость против seed рынка — но из-за H-1 market всегда `lv`, а seed идентичен по всем рынкам, поэтому проверка эффективно всегда против `lv`.
**Почему это проблема:** когда стоимости доставки разойдутся по рынкам, серверная валидация будет опираться на `lv`, а не фактический рынок. Связано с H-1.
**Критичность:** Medium. **Рекомендация:** решается вместе с H-1 (передать корректный market); опционально — читать методы доставки из БД в UI.

# Отдельно: «инфраструктура без использования»

Как требует задание, отмечаю системы, присутствующие только формально:

1. **Market VAT / OSS-атрибуция** — модель и колонка `orders.market_code` есть, но веб-заказ их не заполняет по факту (H-1). Работает только в Storefront API.
2. **market_product_prices (прайс-листы)** — таблица + resolver есть, seed нет, веб-витрина не читает (M-1). Активно только в Storefront API/checkout-internals.
3. **market_shipping_methods** — seed есть, веб-UI не читает (M-3).
4. **parcel_stations LT/EE** — seed есть, потребитель только Storefront API (L-3).
5. **Inventory reservations** — схема + RPC есть, выполнение выключено флагом (M-2).
6. **Storefront API v1** — реально работает (rate-limited, market-aware), но потребителя (mobile app) нет — заявлено как foundation, это честно.

То есть Phase 6, как и Phase 5, доставляет корректно спроектированный фундамент, значительная часть которого не подключена к фактическому веб-потоку. Это не дефект сам по себе (foundation-scope задекларирован), но headline-формулировка «market-aware checkout» этому не соответствует — заказ рынок не учитывает.

# Top Recommendations

1. **H-1: подключить рынок к веб-checkout.** Разрешать market на сервере в `createCheckoutSession` (из cookie `pharm_market`/заголовков через `cookies()`), передавать в `createDraftOrder`. Без этого «международность» не поставлена, а LT/EE-запуск даст неверный НДС/OSS. Наивысший приоритет.
2. **M-1: унифицировать ценообразование** — провести market pricing в веб-витрину или задокументировать API-only статус override-ов; добавить инвариант-тест «display price == charged price» для рынка.
3. **Рантайм-проверка на БД** (я не смог — нет CLI): `supabase db:reset` + сценарии — resolveMarket по geo-заголовку, per-market VAT в заказе, RLS на новых таблицах, Storefront API против реальных данных.
4. **M-2: не включать `INVENTORY_RESERVATIONS_ENABLED` в prod** до полного жизненного цикла (consume/release на paid + cron истечения + гейт checkout).
5. **L-1: закрыть CSP `unsafe-inline`** (nonce-based) до реального трафика — единственный оставшийся security-долг с Phase 2.

# CTO Verdict

Phase 6 — снова дисциплинированная foundation-фаза: закоммичена, отмечена тегом, зелёная во всех прогонах, которые я выполнил лично (tsc 0 / lint 0 errors / tests 44 / build ok). Реальный прогресс: замечания отчёта 20 закрыты не декларативно, а по коду — типы регенерированы (закрыт главный долг Phase 5), мёртвый keyset удалён, дефектные резервации безопасно выключены. Модель рынков спроектирована чисто (детерминированный `resolveMarket`, core/impl-разделение для тестов, корректный RLS и VAT-seed, включая EE 22%).

Но заявка «international foundation» держится на одной несделанной интеграции: **веб-оформление заказа не привязано к разрешённому рынку**, поэтому per-market VAT и OSS-атрибуция — центральные цели фазы — в фактическом потоке заказов не работают и жёстко зашиты в LV. Это дормантно и безвредно при текущем LV-only, но активируется как дефект налоговой корректности ровно в момент запуска LT/EE — то есть при первом же использовании того, ради чего фаза делалась.

**Решение:** для LV-only production — деплоибельно (CONDITIONAL PASS), ничего рабочего не сломано. Для мульти-рыночного запуска — **не готово**: H-1 обязателен к закрытию, плюс рантайм-проверка на реальной БД, которую в этой среде я подтвердить не смог. Зрелость: **B** — крепкий фундамент с одной критичной для целей фазы недостающей связкой. Новый нюанс, повторяющий урок Phase 5: зелёный CI подтверждает, что код собирается и модель корректна, но не что фича подключена к пути пользователя — здесь рынок доходит до middleware и API, но не до заказа.
