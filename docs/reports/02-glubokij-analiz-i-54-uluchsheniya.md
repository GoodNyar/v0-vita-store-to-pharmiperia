# Отчёт 02 — Глубокий анализ проекта: архитектура, сильные стороны, техдолг, 54 улучшения

> Дата: 2026-07-02 · Источник: полное изучение кодовой базы (конфигурация, ядро, страницы, компоненты, SQL-схемы)

## 1. Архитектура проекта

**Стек:** Next.js 16 (App Router) + React 19 + TypeScript (strict) + Tailwind CSS 4 + shadcn/ui (Radix) + Supabase (`@supabase/ssr`) + Stripe (Embedded Checkout) + Vercel AI SDK (чат и рекомендации). Проект сгенерирован через **v0.app** и дорабатывался вручную.

**Ключевой архитектурный факт: два параллельных, несовместимых источника данных о товарах:**

1. **Хардкод `lib/data.ts`** — 16 товаров с числовыми `id` (1–16), одноязычными полями. Используют: главная, страница товара, категории, бренды, popular, specials, избранное, sitemap.
2. **Supabase-схема `scripts/001_create_schema.sql`** — полноценная e-commerce БД с UUID, двуязычными полями (`name_ru`/`name_lv`), стоками, промокодами. Используют: поиск, AI-рекомендации, заказы, отзывы, адреса, профили.

Эти два мира не стыкуются: числовой `id=1` вставляется в UUID-колонки, запросы обращаются к несуществующим колонкам (`active` вместо `is_active`, `name` вместо `name_ru`), таблица товаров в БД пуста (нет seed). Половина функциональности молча не работает.

### App Router и маршрутизация

App Router используется формально: 33 из 40 страниц — `"use client"` целиком. SSR задействован только в `layout.tsx`-обёртках трёх маршрутов (`products/[id]`, `category`, `brand`) — там `generateMetadata`, `generateStaticParams`, JSON-LD (сделано грамотно). Данные грузятся в `useEffect`, навигация местами через `window.location.href` и голый `<a href>` (product-card) — полная перезагрузка.

**Дублирование провайдеров:** глобальный layout оборачивает всё в `LangProvider → AuthProvider → FavoritesProvider → CartProvider → ToastProvider`, но **24 страницы дополнительно оборачивают себя** в собственные `LangProvider`/`CartProvider`. Вложенный провайдер затеняет корневой → расщеплённое состояние.

### Состояние

Четыре контекста: Auth (user/session + onAuthStateChange), Cart (localStorage `pharmiperia_cart`), Favorites (гость — localStorage, юзер — таблица favorites + merge при входе, оптимистичные обновления), Lang (RU/LV, автоопределение, localStorage). Серверного состояния корзины нет — «merge guest cart» merge'ит корзину саму с собой.

### Supabase

- Клиенты по канону `@supabase/ssr`: browser/server/middleware (`lib/supabase/proxy.ts` — обновление сессии + защита `/account/*`).
- Схема БД хорошая: 10 таблиц, индексы, RLS везде, триггер автосоздания профиля (SECURITY DEFINER), триггер пересчёта рейтинга.
- Но клиентский код написан под другую схему: поиск фильтрует по `active`/`name`; заказы вставляются без обязательных `email/first_name/last_name` и с `price` вместо `unit_price`; отзывы — `verified_purchase` вместо `is_verified_purchase`; страница лояльности читает несуществующие `loyalty_points`/`loyalty_transactions`.
- В `lib/supabase/client.ts` — самодельный парсинг `document.cookie` через `split("=")`, обрезающий значения с `=`.

### Stripe

Embedded Checkout: server action `createCheckoutSession` → `<EmbeddedCheckout>` → `onComplete` на клиенте пишет заказ со статусом `'paid'`. Три фундаментальные проблемы:
1. **Цены берутся из клиента** (комментарий «prices validated server-side» лжёт) — можно купить всё за 0,01 €.
2. **Нет вебхука** — факт оплаты фиксируется только клиентским колбэком.
3. Заказ гостя не сохраняется вообще; у залогиненного вставка падает из-за схемы, ошибка глотается.

### Авторизация

Email/пароль + Google OAuth, PKCE-колбэк, защита в middleware через `getUser()` — в целом правильно. Но: «Забыли пароль?» ведёт на несуществующий `/auth/reset-password` (404); redirect на логин не сохраняет `?next=`; профиль (PII) кэшируется в localStorage.

## 2. Сильные стороны

1. Современный, правильно подобранный стек.
2. Качественная схема БД — таблицы, индексы, RLS, триггеры, двуязычные поля.
3. Серьёзная работа над SEO — metadata, OG/Twitter, JSON-LD, sitemap, robots, canonical, generateStaticParams.
4. Полная двуязычность RU/LV (1787 строк переводов), автоопределение языка.
5. Продуманный guest-flow — корзина/избранное без входа + merge при логине, оптимистичный UI.
6. UX-полиш — скелетоны, lazy-загрузка, sticky-шапка, адаптив, aria-атрибуты, дизайн-система на CSS-переменных (+тёмная тема).
7. Каноничная интеграция Supabase SSR, `server-only` для Stripe-секрета.
8. Богатая функциональность для MVP — AI-чат, AI-подбор, отзывы, лояльность, блог, 12+ контентных страниц.

## 3. Технический долг, проблемы, риски

### Сломанная функциональность (баги)

- **`clearCart` не существует** в CartContext, но вызывается в `checkout/page.tsx:137` и `checkout/success/page.tsx:23` → TypeError сразу после успешной оплаты.
- **`addItem` не существует**, но вызывается на страницах поиска и категорий → «В корзину» крашится.
- **Все 16 картинок товаров отсутствуют** (`/images/products/*.jpg` нет в public/). Также нет `og-default.jpg`, `icon.svg`, `apple-icon.png`, favicon.
- **Поиск всегда пуст** — несуществующие колонки `active`/`name`/`description`.
- **Заказы не сохраняются никогда** — NOT NULL нарушения, число в UUID, неверные колонки; ошибки глотаются.
- **Отзывы нельзя отправить** (UUID/колонки); имена авторов не отобразятся из-за RLS `profiles_select_own`.
- **Избранное у залогиненных не пишется в БД** (число в UUID).
- **Лояльность читает несуществующие таблицы.**
- **Битые URL брендов в sitemap** — локальная `brandSlug()` даёт `/brand/avne` вместо `/brand/avene`.
- Товары 13–16 без обязательных `sku`/`volume`/`description` → `#undefined` на карточках.
- Форма подписки на рассылку — без обработчика.

### Архитектурные проблемы

- **`typescript.ignoreBuildErrors: true`** — маскирует все ошибки типов выше.
- **Двойная модель данных** — корень большинства багов.
- Клиентский монолит; `account/page.tsx` — 1319 строк.
- Дублирование: `getBrandSlug` в 4 местах; провайдеры на 24 страницах; два комплекта SQL; два globals.css.
- Мёртвый код: `quick-view-modal`, `pull-to-refresh`, `theme-provider`, `category-sidebar`; большинство ui-компонентов и зависимостей (`recharts`, `react-resizable-panels`, `embla-carousel`, `input-otp`, `vaul`…); шрифты Geist грузятся, но не применяются; `popular.png`/`specials.png`/`tsconfig.tsbuildinfo` в корне.
- Нет инженерной обвязки: не git-репозиторий, нет тестов, нет ESLint (и в devDependencies), нет CI, `.env.example`, `error.tsx`/`not-found.tsx`/`loading.tsx`.

### Риски безопасности

1. Подмена цены при оплате (критично).
2. Фиксация оплаты без верификации (нет вебхука).
3. `/api/chat` и `/api/recommendations` без авторизации и rate-limit — открытый прокси к GPT (выжигание бюджета).
4. RLS promo_codes позволяет анонимно выгрузить все активные промокоды.
5. PostgREST-инъекция в поиске (интерполяция query в `.or()`).
6. PII в localStorage.
7. Нет security-заголовков, нет GDPR cookie-баннера.
8. Гостевой заказ невозможен по RLS при UI, обещающем оплату гостю.

## TLDR

UI-слой сильный (дизайн, i18n, SEO-обвязка, guest-flow), но e-commerce-ядро фактически не работает: каталог с битыми картинками, поиск пуст, «В корзину» на части страниц и экран после оплаты крашатся, заказы не пишутся в БД, оплата доверяет ценам браузера и не подтверждается вебхуком. Всё маскируется `ignoreBuildErrors: true`. Первые шаги — пункты 1–14.
