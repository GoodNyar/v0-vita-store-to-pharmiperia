# Отчёт 07 — Staff-Level Code Review

> Дата: 2026-07-03 · Формат: ревью кода «как перед наймом Staff-инженера» · Только качество кода.
> Находки отчётов 02–06 **не повторяются** — здесь только новые наблюдения. Ранги: **S** критично · **A** высоко · **B** средне · **C** низко · **D** мелочь.

**Вердикт нанимающего:** кандидат демонстрирует хороший продуктовый вкус и знание экосистемы по верхам, но код показывает системную привычку: **писать против воображаемых контрактов, не проверяя реальные** (API библиотек, соседние словари статусов, собственные утилиты). На Staff-уровень это не проходит: Staff-инженер отличается именно тем, что его код честен с зависимостями. Оценка: сильный Mid / слабый Senior. Ниже — 25 находок с обоснованием.

---

## 1. Библиотечные контракты (SDK drift)

### S-1. Чат написан под удалённый API AI SDK — виджет неработоспособен
- **Объяснение:** `components/live-chat.tsx:14` деструктурирует `{ input, setInput, initialMessages }` из `useChat`. В установленных `@ai-sdk/react@^3` / `ai@^6` управление полем ввода удалено из `useChat` (миграция AI SDK 5.0: `input`/`setInput`/`handleSubmit` убраны; `initialMessages` переименован в `messages`). `input === undefined` → `value={input}` (строка 162) даёт неуправляемый input, `!input.trim()` в `disabled` (строка 170) **бросает TypeError прямо в рендере** при открытии чата; приветственное сообщение не появляется.
- **Файлы:** `components/live-chat.tsx`.
- **Почему важно:** это витринная фича («живой консультант»), падающая при первом клике; ошибка типов существует, но спрятана `ignoreBuildErrors`. Показательно: код писался под версию из головы, а не из `package.json`.
- **Решение:** локальный `useState` для поля ввода + `sendMessage({ text })`; приветствие — через `messages`-опцию; включённый typecheck (blueprint A1) не даст этому классу ошибок появляться.

### B-2. `generateText` с параметром `output:` — API под вопросом для установленной мажорной версии
- **Объяснение:** `app/api/recommendations/route.ts:72` передаёт `output: Output.object({...})`. В AI SDK 5 этот параметр называется `experimental_output`; сигнатура для `ai@6` требует сверки — если параметр игнорируется, `result.object` будет `undefined` и рекомендации молча пустые. Рядом D-24: `await convertToModelMessages(...)` — await на синхронной функции (`app/api/chat/route.ts:29`).
- **Файлы:** `app/api/recommendations/route.ts`, `app/api/chat/route.ts`.
- **Почему важно:** тот же паттерн, что S-1 — код против воображаемой сигнатуры; сбой тихий.
- **Решение:** для структурированного вывода использовать `generateObject({ model, schema, prompt })` — прямой, стабильный API; smoke-тест эндпоинта в CI.

## 2. React-паттерны и state management

### A-3. Ни один context value не мемоизирован — заявленные оптимизации не работают
- **Объяснение:** все пять провайдеров передают value инлайн-объектом (`value={{...}}`), `useMemo` — 0 вхождений на все провайдеры; в `LangProvider` функция `t` пересоздаётся каждый рендер. Значит, identity контекста меняется при каждом рендере провайдера, и `memo(ProductCardComponent)` (`components/product-card.tsx:162`) не срабатывает никогда: `useCart`/`useLang`/`useFavorites` всегда «новые».
- **Файлы:** `components/cart-context.tsx`, `components/auth-provider.tsx`, `components/favorites-provider.tsx`, `components/toast-provider.tsx`, `lib/i18n.tsx`.
- **Почему важно:** это не микрооптимизация — это противоречие внутри кода: автор добавил `memo`, не проверив, что его гарантии разрушены этажом выше. На ревью такое читается как карго-культ.
- **Решение:** `useMemo` на value, `useCallback` на функции контекста, стабильный `t` (зависимость только от `lang`); правило Playbook: контекст-value всегда мемоизирован.

### A-4. Ротация промо-бара ре-рендерит весь хедер каждые 4 секунды
- **Объяснение:** `promoIndex`-состояние живёт в корне `SiteHeader` (`components/site-header.tsx:57-67`) — интервал каждые 4 с обновляет state, ре-рендеря шапку целиком (логотип, поиск, навигацию с дропдаунами, счётчики) на каждой странице; в связке с A-3 волна идёт дальше.
- **Файлы:** `components/site-header.tsx`.
- **Почему важно:** фоновая работа CPU на каждой странице сайта; шаблон «состояние выше, чем нужно».
- **Решение:** выделить `<PromoBar/>` с собственным состоянием; заодно распилить 507-строчный файл на PromoBar/MainBar/CategoryNav/MobileSidebar.

### A-5. `FavoritesProvider`: гонки и восстановление устаревшего снимка
- **Объяснение:** `toggleFavorite` (`components/favorites-provider.tsx:126-162`) читает и «откатывает» состояние из замыкания (`setFavorites(favorites)`), а не функциональным апдейтом: два быстрых клика по разным товарам → второй откат затирает первый; insert в БД без `onConflict` при `UNIQUE(user_id, product_id)` → повторный клик в гонке даёт ошибку дубликата, которая проглатывается.
- **Файлы:** `components/favorites-provider.tsx`.
- **Почему важно:** классическая stale-closure ошибка — маркер непонимания модели состояния React; у избранного будет «мигающее» поведение под реальными кликами.
- **Решение:** функциональные апдейты `setFavorites(prev => ...)`, `upsert(..., { onConflict: 'user_id,product_id', ignoreDuplicates: true })`, откат через пересинхронизацию с БД.

### B-6. Семантика количества через цикл добавлений
- **Объяснение:** `handleAddToCart` (`app/products/[id]/page.tsx:35-39`) вызывает `addToCart(product)` в цикле `quantity` раз — N последовательных обновлений состояния и N записей в localStorage; сам `addToCart` побочно открывает drawer (решение UI зашито в доменный метод).
- **Файлы:** `app/products/[id]/page.tsx`, `components/cart-context.tsx`.
- **Решение:** `addCartLine(product, qty)` (словарь ADR-0004/blueprint B5); открытие drawer — на вызывающей стороне.

### B-7. `AuthProvider`: двойной сетевой вызов и мёртвый параметр
- **Объяснение:** `initAuth` последовательно вызывает `getUser()` **и** `getSession()` (два обращения при монтировании каждого дерева), хотя ниже подписка `onAuthStateChange` всё равно перезапишет состояние; проп `initialSession` не передаётся ни одним потребителем — ветка кода мёртвая.
- **Файлы:** `components/auth-provider.tsx`.
- **Решение:** одно чтение + подписка; либо реально передавать `initialSession` с сервера (тогда исчезает и «мигание» isLoading).

## 3. Дублирование и мёртвый код (новые случаи)

### A-8. Два «экрана благодарности» и два генератора номеров заказа
- **Объяснение:** пост-оплатный UX реализован дважды: inline-шаг `complete` в `app/checkout/page.tsx:163-199` и страница `app/checkout/success/page.tsx` — на которую **нет ни одной ссылки** (Stripe-сессия создаётся с `redirect_on_completion: 'never'`), т.е. маршрут недостижим. Каждая версия генерирует свой фиктивный номер заказа: `PH${Date.now().toString(36).toUpperCase()}` против `PH${Date.now().toString().slice(-8)}`.
- **Файлы:** `app/checkout/page.tsx`, `app/checkout/success/page.tsx`.
- **Почему важно:**两 реализации одного экрана разойдутся (уже разошлись форматом номера); недостижимый маршрут — ловушка для следующего разработчика.
- **Решение:** одна серверная страница подтверждения, читающая заказ из БД (естественно закрывается blueprint C2); номер заказа генерирует только сервер.

### B-9. Четыре системы тостов в одном проекте
- **Объяснение:** (1) собственный `components/toast-provider.tsx` (контекст, подключён в layout), (2) shadcn-пара `hooks/use-toast.ts` + `components/ui/toaster.tsx` (не подключена), (3) `sonner` в зависимостях + `components/ui/sonner.tsx`, (4) рукописный тост с позиционированием по координатам курсора в `app/account/page.tsx:88-94`.
- **Файлы:** перечислены выше.
- **Решение:** одна система (sonner — самая простая и уже в deps), остальные три удалить; курсорные координаты для тостов — анти-паттерн доступности.

### B-10. Два компонента `Skeleton`
- **Объяснение:** `components/loading-skeleton.tsx` и `components/ui/skeleton.tsx` — одинаковая функциональность (div + animate-pulse + bg-muted), разные импорты по коду.
- **Решение:** оставить `ui/skeleton`, обновить импорты.

### B-11. Звёздный рейтинг написан заново в 9 файлах
- **Объяснение:** рендер звёзд продублирован в `product-card`, `products/[id]`, `category/[slug]`, `search`, `reviews`, `account/favorites`, `account/page`, `product-reviews`, `quick-view-modal` — с разнобоем: где-то `fill-amber-400`, где-то `fill-yellow-400`, где-то полные 5 звёзд, где-то одна звезда + число.
- **Почему важно:** канонический тест на абстракции: повторяемый визуальный примитив без компонента. Изменение дизайна рейтинга = 9 правок.
- **Решение:** `components/product/rating.tsx` (`<Rating value reviewCount size />`), один токен цвета.

### B-12. shadcn установлен, формы — рукописные
- **Объяснение:** input-класс длиной ~200 символов (`w-full rounded-lg border ... focus:ring-primary/20`) повторён 10+ раз в `checkout/page.tsx`, по нескольку раз в `auth/login`, `auth/sign-up`, `contact`, `account` — при существующем `components/ui/input.tsx`; кнопки — местами `ui/button`, местами сырые `<button>` с теми же классами.
- **Почему важно:** дизайн уже дрейфует (фокус-стили отличаются между формами); каждое изменение темы = массовые правки строк.
- **Решение:** `ui/input` + `ui/label` + вариант ошибки через `cva`; форма чекаута — `react-hook-form` (установлен и не используется — ещё одна «зависимость-призрак»).

### B-13. Мёртвый и при этом публичный server action
- **Объяснение:** `getCheckoutSession` (`app/actions/stripe.ts:61-69`) не вызывается нигде, но server actions компилируются в публичные POST-эндпоинты: неаутентифицированный вызов с известным session id вернёт email покупателя и сумму. Session id неугадываем, поэтому риск низкий — но это неиспользуемая публичная поверхность с PII.
- **Решение:** удалить; если понадобится — верифицировать принадлежность сессии текущему пользователю.

## 4. TypeScript-качество

### A-14. Расходящиеся доменные словари в соседних файлах
- **Объяснение:** `interface Order` определён дважды с несовместимыми статусами: `app/account/page.tsx:44` — `"delivered" | "in_progress" | "cancelled"`; `app/account/orders/page.tsx:23` — `status: string` с обработкой `'paid' | 'processing' | 'shipped' | 'delivered'`. `interface Profile` определён дважды (`account/page.tsx`, `account/settings/page.tsx`) с разными полями. `interface Product` — дважды (`category`, `search`) поверх третьего в `lib/data.ts`.
- **Почему важно:** это глубже, чем «рукописные типы» (отчёт 02): два экрана одного раздела **семантически** не согласны, что такое статус заказа. Любой рефакторинг статусов сломает один из них молча.
- **Решение:** доменные типы — только из `lib/commerce` поверх сгенерированных (blueprint B2/B3); `OrderStatus` — один union + один маппинг на переводы/иконки.

### B-15. `select('*')` + `as any` на границах данных
- **Объяснение:** пять запросов читают `select('*')` (`account/loyalty:106,117`, `account/addresses:55`, `account/orders:68`, `product-reviews:90`) с последующим `setX(data as any)` — максимальный over-fetch ровно там, где нужен контракт; `as any`-каст закрывает единственное место, где типы могли бы поймать дрейф схемы.
- **Решение:** явные списки колонок + типы из `supabase gen types`; ESLint-запрет `as any` (Playbook §3).

### C-16. Конфигурация компилятора устарела
- **Объяснение:** `tsconfig.json`: `target: "ES6"` (2015) при Next 16/React 19 — лишняя транспиляция вниз; `allowJs: true` без единого .js-файла; нет `noUncheckedIndexedAccess` (индексные обращения вроде `promoItems[promoIndex]` не проверяются — рядом уже живёт защитный `currentPromo?.icon`, симптом).
- **Решение:** `target: "ES2022"`, убрать `allowJs`, включить `noUncheckedIndexedAccess` (после A1 blueprint).

## 5. Naming и читаемость

### B-17. Импорт в середине файла
- **Объяснение:** `components/site-header.tsx:44` — `import { Button } ...` расположен **после** констант и функций (строки 24–43). Код исполняется (hoisting), но это след бессистемных правок и отсутствия линтера.
- **Решение:** Prettier + `import/order` в ESLint (blueprint A1); показательный маркер для §14 Playbook (артефакты генерации).

### C-18. Словарь переводов: четыре ключа об одном
- **Объяснение:** `bestSeller`, `bestSellers`, `bestsellers`, `bestSellersSection` сосуществуют в `lib/i18n.tsx` (строки 24, 37, 84, 91) с почти одинаковыми значениями; ключи-нумерации `checkoutSteps1/2`, `productBenefit1..5` кодируют контент порядковым номером.
- **Решение:** закроется namespace-схемой ADR-0007; добавить скрипт «неиспользуемые ключи словаря» в CI.

### C-19. Магические значения россыпью
- **Объяснение:** `setTimeout(..., 1800)` (тост), `bonusMax = 200`, интервал промо `4000`, `style={{ maxHeight: "calc(100vh - 220px)" }}` (число 220 связано с высотой хедера неявно), `slice(0, 8)` на главной.
- **Решение:** именованные константы рядом с владельцем поведения; для высоты — CSS-переменная от хедера.

## 6. Контент и данные в коде

### C-20. Один состав INCI захардкожен для всех товаров
- **Объяснение:** таб «Состав» на странице любого товара показывает один и тот же список ингредиентов с экстрактом огурца (`app/products/[id]/page.tsx:42-45`, `productContent`), от мицеллярной воды до шампуня. Для аптечной косметики состав — фактор покупки и юридическая информация.
- **Почему важно:** фиктивные данные, неотличимые от реальных, хуже отсутствующих (принцип «фейковых фич», Playbook §17.10).
- **Решение:** поля `ingredients`, `how_to_use_*` уже есть в схеме БД — таб рендерить только при наличии данных (закрывается blueprint B6).

## 7. Stripe-интеграция (паттерны кода)

### C-21. Нестабильные options у EmbeddedCheckoutProvider
- **Объяснение:** `checkout/page.tsx:416-421` строит `items.map(...)` инлайн в JSX → новый массив каждый рендер → `useCallback([items, ...])` в `stripe-checkout.tsx:35-46` пересоздаёт `fetchClientSecret` на каждый рендер. `react-stripe-js` игнорирует смену options после инициализации, поэтому не стреляет — но паттерн хрупок: любое будущее «пересоздать сессию при смене корзины» сработает хаотично. Плюс `loadStripe` на module scope грузит Stripe.js уже на шаге «данные», до шага оплаты.
- **Решение:** `useMemo` для items; `next/dynamic` для StripeCheckout — монтировать на шаге payment.

## 8. Мелочи (D)

### D-22. JSON-LD без экранирования `<`
- `JSON.stringify` в `dangerouslySetInnerHTML` (`products/[id]/layout.tsx`, `org-json-ld.tsx`) не экранирует `<` — имя товара, содержащее `</script>`, разорвёт документ. Данные админские, риск низкий. Решение: `.replace(/</g, '\\u003c')`.

### D-23. `key={index}` в списках
- `checkout/success` steps, thumbnail-плейсхолдеры товара. Безвредно для статических списков, но против конвенции.

### D-24. `await` на синхронной функции
- `app/api/chat/route.ts:29` — см. B-2.

### D-25. Захардкоженная локаль форматирования
- `reviewCount.toLocaleString("ru")` в `product-card.tsx:120` при доступном `lang` в той же области видимости; `formatEur` фиксирует `de-DE` (закрывается ADR-0002/0007). Плюрализация «product/products/productsLabel» тернарниками в `account/favorites/page.tsx:58`.

## «Если бы я унаследовал этот проект сегодня — что я изменил бы за первые 30 рабочих дней»

Дни рабочие; ссылки — на шаги blueprint (отчёт 06). Ничего из этого не требует переписывания — только последовательность мелких, откатываемых PR.

**Дни 1–3 — сделать проект честным.**
`git init` + коммит as-is; CI с `tsc --noEmit`, ESLint (+ `import/order`, запрет `as any`), Prettier по всей базе одним коммитом; убрать `ignoreBuildErrors` и починить всё, что вспыхнет — включая S-1 (чат), фантомные `clearCart`/`addItem` и товары 13–16. К вечеру третьего дня компилятор снова мой коллега, а не заглушенный свидетель. *(A1, A2 частично)*

**Дни 4–5 — убрать двойников.**
Одна система тостов (sonner), один Skeleton, один экран «спасибо» (success-маршрут удаляю), `getCheckoutSession` удаляю, мёртвые компоненты и зависимости — под нож; `<Rating/>` компонент вместо девяти копий. Диффы большие, риск нулевой, кодовая база худеет процентов на десять. *(A2, B-8..B-13 этого отчёта)*

**Дни 6–8 — починить фундамент состояния.**
