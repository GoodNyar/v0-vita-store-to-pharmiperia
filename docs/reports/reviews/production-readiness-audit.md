# Отчёт 04 — Production Readiness Audit

> Дата аудита: 2026-07-02 · Цель: запуск через 2 недели · Вердикт: **NO-GO**

## 1. Production Readiness
**Хорошо:** managed-платформа (TLS/CDN/автодеплой); юридические страницы наполнены. **Отсутствует:** домен; БД пуста (нечего продавать); контактная форма — имитация (`setTimeout`), трекинг всегда «не найдено», рассылка no-op; блокеры денежного пути. **До запуска:** домен; наполнить БД; честно отключить фичи-пустышки. **После:** трекинг перевозчиков, рассылка.

## 2. Reliability
**Хорошо:** serverless HA; корзина переживает перезагрузку; оптимистичные обновления. **Отсутствует:** ни одного `error.tsx`/`global-error.tsx`/`not-found.tsx`/`loading.tsx`; нет таймаутов/ретраев; вечные спиннеры при недоступности БД. **До:** error boundaries, деградация AI-виджетов. **После:** ретраи, circuit breakers.

## 3. Observability
**Есть:** только Vercel Analytics. **Отсутствует:** Sentry, structured logs, видимость ошибок оплаты. **До:** Sentry (client+server), алерт на ошибки checkout. **После:** трейсинг, SLO. Критичность: блокер — «слепой» денежный путь.

## 4. Monitoring
**Отсутствует полностью:** uptime, синтетический чекаут, health-endpoint, алерты квот (Supabase, AI-бюджет), Stripe-алерты. **До:** внешний uptime-монитор, бюджет-алерты. **После:** синтетический e2e по крону.

## 5. Logging
`console.log("[v0]")` в проде; ошибки заказов — в консоль браузера покупателя; нет серверного журнала заказа. **До:** убрать debug-логи, логировать server actions. **После:** аудит-лог событий заказа.

## 6. Error Handling
**Хорошо:** валидация форм, error-состояние Stripe-компонента. **Отсутствует:** глотание ошибок на деньгах (успех при провале записи заказа); «пусто» неотличимо от «сломалось». **До:** различать пусто/ошибка; никогда не показывать успех без подтверждения записи.

## 7. Backup Strategy
**Хорошо:** схема как код (scripts/*.sql); Stripe — независимая копия платёжных фактов. **Отсутствует:** план Supabase неизвестен (Free — без бэкапов), тест восстановления не проводился. **До:** Supabase Pro + бэкапы + один тест восстановления. **После:** PITR, квартальные проверки.

## 8. Disaster Recovery
**Отсутствует:** RTO/RPO, runbook'и, процедура ротации ключей. **До:** одностраничный runbook (ключи, ротация, «платежи падают»). **После:** формальные RTO/RPO, учения. Критично: процедура ротации Stripe-ключа до первого платежа.

## 9. Deployment
**Хорошо:** v0→Vercel автодеплой, preview, rollback. **Отсутствует:** локального git нет; staging нет (эксперименты на проде); миграции руками; `ignoreBuildErrors` пропускает сломанный код. **До:** git, staging (отдельный Supabase + Stripe test), миграции через CLI.

## 10. CI/CD
**Отсутствует полностью** (typecheck отключён, ESLint не установлен, тестов нет). **До:** pipeline: tsc → eslint → build; включить ошибки сборки. **После:** e2e-смоук на preview.

## 11. Testing Strategy
Ноль тестов. **До запуска:** один Playwright happy-path (каталог→корзина→оплата→заказ), юнит-тесты денежной арифметики, ручная матрица RU/LV × mobile/desktop × гость/юзер. **После:** регресс, визуальные, нагрузочные.

## 12. Performance
**Хорошо:** lazy below-fold, скелетоны, memo, лёгкие страницы. **Отсутствует:** `unoptimized: true` (LCP-провал на мобильных), i18n-словарь в каждом бандле, middleware-latency на всех страницах, мёртвые зависимости. **До:** включить оптимизацию изображений, Lighthouse на 4G. **После:** code-splitting словарей, кэширование.

## 13. Security Hardening
**Хорошо:** RLS, server-only Stripe key, PKCE. **Отсутствует:** security-заголовки (CSP, HSTS, XFO…), rate limiting как класс, captcha, парольная политика 6 символов, полноправный Stripe-ключ, pnpm audit. **До:** заголовки, rate limit, captcha, restricted key, audit в CI. **После:** пентест, security.txt.

## 14. GDPR
**Хорошо:** privacy/terms есть; сбор минимален; VA cookieless. **Отсутствует:** удаление аккаунта (ст. 17), экспорт (ст. 15/20), **AI-чат шлёт сообщения в OpenAI — процессор не раскрыт в политике**, регион Supabase не подтверждён EU, PII в localStorage, cookie-consent для Stripe-cookies, сроки хранения. **До:** EU-регион, политика с реальными процессорами (Vercel, Supabase, Stripe, OpenAI), процедура удаления/экспорта, consent-баннер. **После:** автоматизация DSAR. Критичность: блокер (продажа физлицам в ЕС).

## 15. SEO Readiness
**Хорошо:** лучший слой проекта — metadata, JSON-LD, sitemap/robots (правильные disallow), generateStaticParams. **Отсутствует:** **canonical в корневом layout наследуется всеми страницами → «весь сайт = дубликат главной»**; OG/фавиконки физически отсутствуют; hreflang нет; Search Console не подключён; битые brand-URL в sitemap. **До:** canonical per-page, реальные OG/иконки, Search Console, www-редирект. **После:** hreflang, контент-стратегия.

## 16. Accessibility
**Хорошо:** aria-label, htmlFor, focus-кольца, Radix-примитивы, клавиатурный обработчик в карточке. **Отсутствует:** корзина/меню — самодельные div без focus trap/Escape/aria-modal; выпадающие меню — hover-only; aria-live для тостов; `<html lang>` статичен. Контекст: European Accessibility Act действует с июня 2025 (микропредприятия исключены). **До:** focus trap + клавиатурные меню. **После:** полный аудит axe.

## 17. Analytics
**Есть:** Vercel Analytics (pageviews, Web Vitals). **Отсутствует:** e-commerce-события (view_item, add_to_cart, begin_checkout, purchase), воронка, UTM, серверный purchase, запись поисковых запросов. **До:** минимум add_to_cart + purchase. **После:** полная воронка. Критичность: высокая — маркетинг вслепую.

## 18. Payments
**Хорошо:** Stripe Embedded (PCI-scope, 3DS/SCA), EUR, customer_email. **Отсутствует:** вебхук (сверка невозможна); **НДС (PVN 21%) не существует в системе** — юридический блокер; возвраты не предусмотрены при «14 дней» в оферте; **футер рекламирует Apple/Google Pay и банклинки SEB/Swedbank/Citadele/Luminor/Revolut, которых нет в чекауте** (а банклинки — доминирующий метод в Латвии); test→live чеклист не пройден. **До:** live-активация, Stripe Tax, вебхук, реальные латвийские методы, процесс возврата. **После:** автоинвойсы, диспуты.

## 19. Email Infrastructure
**Есть:** auth-письма Supabase. **Отсутствует:** встроенный SMTP Supabase шлёт с supabase.io с лимитом единиц писем/час; кастомный SMTP нет; SPF/DKIM/DMARC невозможны (домена нет); письмо заказа не существует при UI-тексте «письмо отправлено»; support@ не может существовать. **До:** домен → почта → support@ → кастомный SMTP (Resend/Postmark) + SPF/DKIM/DMARC → письмо заказа. **После:** маркетинг, DMARC p=reject.

## 20. Domain & DNS
**Отсутствует сам домен** — pharmiperia.lv свободен (проверено whois). Открытый риск: имя опубликовано, домен может занять кто угодно. **До (день 1):** регистрация .lv (+ .com, опечатки), NS, www-редирект, CAA, автопродление. Критичность: максимальная во всём аудите.

## 21. Infrastructure
**Хорошо:** полностью managed-стек. **Отсутствует:** `.env.example` и инвентарь переменных; планы/регионы не зафиксированы (Supabase Free приостанавливает неактивные проекты); конфигурация auth/SMTP не задокументирована. **До:** Supabase Pro (EU), документ окружений, `.env.example`.

## 22. Scalability (масштаб запуска)
Для стартового трафика стек выдержит с запасом. Две точки риска: auth-запрос middleware на каждый pageview (лимиты Supabase Auth) и безлимитные AI-эндпоинты. **До:** rate limit `/api/*`. **После:** кэширование.

## 23. Cost Optimization
**Хорошо:** стартовая стоимость почти нулевая. **Риск №1: AI-эндпоинты — расходы без потолка** (аноним + скрипт = произвольный счёт; recommendations шлёт весь каталог в промпт). **До:** hard budget cap на AI + алерты 50/80/100%, Vercel spend controls. **После:** кэш ответов, egress.

## Итоговое заключение

Блокеры в трёх группах:
1. **Существование бизнеса:** домен не зарегистрирован → нет сайта, почты, support@, SEO.
2. **Деньги и закон:** НДС отсутствует, заказы не фиксируются (нет вебхука), заявленные методы оплаты ≠ реальные, GDPR не выполнен, транзакционные письма — фикция.
3. **Слепота:** ни error tracking, ни алертов, ни conversion-событий.

Критерий готовности к запуску: **тестовый покупатель проходит путь от карточки до письма с подтверждением, и заказ виден в БД и Stripe с корректным НДС.** Пока сценарий не выполняется — продавать нельзя.
