## Принципы

1. **Необратимое — вперёд.** URL-схема, денежный тип, модель каталога, домен — дорожают экспоненциально. Обратимое (админка, поиск, дизайн) может подождать.
2. **Выручка финансирует рефакторинг.** Магазин не останавливается на «переписывание»; каждая фаза даёт бизнес-результат.
3. **Не строить то, для чего нет спроса.** Поиск до 1000 SKU, мультивалютность до второй страны, приложение до повторных покупок — сжигание ресурсов. Крупные инвестиции привязаны к бизнес-триггерам.

Команда: старт 1–2 fullstack, к Phase 4 — 3–4. Трудоёмкость в человеко-неделях (чн).

## Phase 1 — До первого запуска (недели 0–3)

**Цели:** юридически и финансово честный магазин; зафиксированы необратимые решения; видим ошибки раньше клиентов.

**Задачи:**
1. Домен pharmiperia.lv (+ .com), DNS, почта, SPF/DKIM/DMARC, support@ — день 1.
2. Git вне v0, CI (typecheck+lint+build), убрать ignoreBuildErrors, staging (отдельный Supabase + Stripe test).
3. Денежный путь: серверная валидация цен, Stripe webhook как источник факта оплаты, серверный заказ, починка чекаута, гостевые заказы.
4. Необратимые решения: слаг-URL, локаль в пути (/lv, /ru), деньги в центах + валюта, каталог в Supabase (миграция + seed).
5. Stripe Tax / НДС 21%, live-активация, реальные способы оплаты.
6. Sentry + error boundaries + алерты чекаута; uptime; budget caps (AI/Vercel/Supabase).
7. GDPR-минимум: EU-регион, политика с процессорами (вкл. OpenAI), удаление аккаунта, consent.
8. Письмо заказа (Resend), кастомный SMTP auth.
9. Контент: картинки товаров, OG/фавиконки, canonical per-page, отключение фиктивных фич.
10. Rate limit + captcha; AI-рекомендации выключить до Phase 3, чат за лимитом.
11. Один Playwright-тест: каталог → корзина → оплата → заказ в БД.

**Почему сейчас:** смена URL и типа денег стоит дни, а не кварталы; закон и деньги. **Анти-скоуп:** админка, поисковый движок, редизайн, приложение, лояльность, тотальная RSC-миграция. **Трудоёмкость:** ~6–7 чн.
**Влияние:** бизнес — право существовать; архитектура — четыре опоры; SEO — правильные URL с первого дня; безопасность — закрыты денежные дыры; производительность — изображения.

## Phase 2 — Первый месяц после запуска (недели 4–8)

**Цели:** воронка в цифрах; операционная зрелость; фундамент админки и API.

**Задачи:** аналитика (GA4/PostHog, view_item/add_to_cart/begin_checkout/purchase — purchase серверно из вебхука, UTM, запись поисковых запросов) · полный набор транзакционных писем (2 языка) · банковские линки Латвии в чекауте · процесс возвратов · **data-access-слой lib/commerce** (все запросы из компонентов → типизированные репозитории, supabase gen types) · Admin v0 (RLS-роль admin + /admin: заказы, статусы, сток) · декремент стока в вебхуке · синтетический e2e по крону · Search Console, контентные страницы → RSC, hreflang lv/ru · a11y quick wins (focus trap, клавиатурные меню) · чистка мёртвого кода, ESLint в CI.

**Почему сейчас:** первый месяц — единственный шанс дёшево учиться на реальном трафике; data-слой до роста кодовой базы. **Зависимости:** purchase ← вебхук; admin ← data-слой. **Трудоёмкость:** ~10 чн.
**Влияние:** воронка видна; банклинки — +15–30% к конверсии оплаты в Балтии; архитектура — API-граница де-факто; SEO — контент ранжируется, hreflang; безопасность — модель ролей.

## Phase 3 — До 10 000 клиентов (месяцы 2–5) ✅ ЗАКРЫТА

> **Tag:** `v3.0-phase3-complete` · **Отчёт:** [phase-3-final-summary.md](phase-3-final-summary.md) · **30/30 PR**

**Цели:** машина удержания; серверная витрина; админка-продукт; события.

**Задачи:** RSC/ISR-миграция витрины (tag-based revalidation; middleware-auth только на защищённых маршрутах) · серверная корзина (merge, цены при чтении) · событийный каркас (Inngest/Supabase Queues): order.paid → письмо/сток/лояльность/аналитика; идемпотентность · **брошенные корзины + welcome-серия + просьба об отзыве** (самые доходные automation) · промо-движок (RPC-валидация, кампании в админке) · лояльность по-настоящему (начисление в вебхуке) · отзывы verified purchase · Admin v1 (CRUD товаров, заказы, промо, audit log) · поиск v1 (pg_trgm + tsvector + фасеты — до 5–10k SKU достаточно) · AI v2 (retrieval/pgvector, бюджеты, кэш) · e2e-набор в CI · helpdesk.

**Почему сейчас:** удержание дешевле привлечения когда появилась база; RSC-миграция ещё дёшева (40 страниц); событийный каркас до обрастания костылями. **Трудоёмкость:** ~30–36 чн.
**Влияние:** LTV-механика (брошенные корзины ≈ +5–10% выручки); архитектура становится трёхслойной (витрина/ядро/события); SEO — серверный HTML каталога = главный скачок; безопасность — идемпотентность, RBAC, audit; производительность — ISR = кэш-хиты, TTFB/LCP падают в разы.

## Phase 5 — До 100 000 SKU (месяцы 5–9) 🔜 ОЖИДАЕТ ТРИГГЕР

> **Baseline:** `v4.0-phase4-complete` · **Prerequisites:** [phase-4-prerequisites.md](phase-4-prerequisites.md)

**Триггер входа:** подписан крупный поставщик с фидом. Без этого фаза не начинается.

**Задачи:** миграция модели каталога (variants, таблицы переводов, атрибуты: тип кожи/SPF/ингредиенты) · импорт-конвейер mini-PIM (фиды CSV/XML → валидация → дедуп → staging → публикация) · медиа: object storage + CDN + деривативы (реальная product_images) · выделенный поиск (Meilisearch/Typesense: опечатки, синонимы, LV/RU-морфология, фасеты, мерчандайзинг) · программный SEO («категория × бренд × свойство»), sitemap-индекс (шарды по 45k URL) · inventory reservations · ревизия индексов/keyset-пагинация/нагрузочный тест · мерчандайзинг в админке.

**Почему сейчас:** модель каталога меняется до массового наполнения — 500 товаров мигрируются тривиально, 100 000 — квартал с даунтаймом. **Зависимости:** всё ← Admin v1 + data-слой; программный SEO ← RSC/ISR. **Трудоёмкость:** ~48–56 чн.
**Влияние:** ассортимент как преимущество, «фид → витрина» за часы; каталог — самостоятельный домен; SEO ×100 страниц + long-tail; валидация фидов; поиск уходит из Postgres.

## Phase 6 — Международный e-commerce (месяцы 9–12+)

**Триггер входа:** стабильная юнит-экономика в Латвии + заметный трафик LT/EE.

**Задачи:** сущность **Market** (страна → валюта, локали, доставка, налоги, оплата, юрлицо) · мультивалютность (прайс-листы поверх минорных единиц, Stripe multi-currency) · **VAT OSS** · i18n-зрелость (lt/et/en, TMS, локализованные sitemap, hreflang-матрица) · перевозчики LT/EE (API Omniva/DPD с картами пакоматов) · **публичный версионированный Storefront API** поверх commerce-ядра · **мобильное приложение** (React Native/Expo поверх API: каталог, корзина, push) · edge/geo-routing · наблюдаемость per-market.

**Почему сейчас:** Балтия — естественная экспансия; ядро событийное и серверное → «рынок» добавляется конфигурацией. **Что нельзя раньше:** приложение до API = вторая кодовая база; мультивалютность до Money-типа = переписывание; экспансия до юнит-экономики = экспорт убытков. **Трудоёмкость:** ~60+ чн.
**Влияние:** TAM ×3–4; приложение = удержание без платного трафика; финальная форма: ядро + N витрин + M рынков; органика в трёх странах.
