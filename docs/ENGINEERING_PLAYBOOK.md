# Pharmiperia Engineering Playbook

**Версия 1.0 · Статус: действующий · Владелец: Engineering**

Этот документ — техническая конституция проекта. Его читает каждый инженер до первого коммита. Изменения в Playbook проходят через PR с ревью двух инженеров. Слова **MUST / MUST NOT / SHOULD / MAY** — в смысле RFC 2119: MUST — нарушение блокирует merge; SHOULD — отступление требует письменного обоснования в PR.

---

## 1. Архитектурные принципы

Итог утверждённых аудитов (см. `docs/reports/`). Не обсуждаются в рамках фичи; пересмотр — только через ADR (§16).

1. **Сервер — источник истины.** Цена, сток, статус заказа, факт оплаты, права доступа вычисляются только на сервере. Клиент — проекция состояния, не его владелец. Данные из браузера считаются враждебными.
2. **Одна модель данных.** Каталог, заказы, пользователи живут в Supabase (Postgres). Второго источника данных о товарах не существует. Константы в TS — только UI-конфигурация, не коммерческие сущности.
3. **Границы слоёв.** Витрина → `lib/commerce` (data-access) → БД. Компоненты не знают о существовании Supabase-клиента. Нарушение границы — дефект, не стиль.
4. **Server-first рендеринг.** Страница — Server Component по умолчанию. `"use client"` — осознанное исключение для интерактивных островков.
5. **События вместо оркестровки в обработчиках.** `order.paid` порождает письмо, сток, лояльность как независимых подписчиков. Побочные эффекты не живут в UI-колбэках.
6. **Необратимое проектируется, обратимое итерируется.** URL, денежный тип, схема БД — через ADR и ревью. Цвет кнопки — через PR без церемоний.
7. **Деньги — целые числа.** Все суммы — в минорных единицах (центах) с явным кодом валюты. Float для денег не существует в кодовой базе.
8. **Локаль — часть URL.** `/lv/...`, `/ru/...` — то, что видит и пользователь, и Google. Язык не хранится в состоянии клиента как источник истины.
9. **Наблюдаемость — часть фичи.** Код, о сбое которого нельзя узнать из Sentry/логов, не готов к продакшену.
10. **Удаляй, не комментируй.** Мёртвый код, неиспользуемые зависимости и «пусть полежит» запрещены. Git помнит всё.

## 2. Структура директорий

```
app/
  [locale]/                # lv | ru — сегмент локали, обязателен для витрины
    (shop)/                # витрина: layout с шапкой/футером
      page.tsx             # главная
      products/[slug]/     # товар (слаги, не ID)
      category/[slug]/
      brand/[slug]/
      search/
      cart/  checkout/
    (content)/             # блог, доставка, юридические страницы (статика/ISR)
    (account)/account/     # кабинет (динамика, auth-only)
    auth/
  api/
    webhooks/stripe/       # единственная точка истины об оплате
    ...                    # публичные endpoints: только тонкие адаптеры к lib/
  admin/                   # админка (RBAC, вне локали)
  sitemap.ts  robots.ts
lib/
  commerce/                # ЕДИНСТВЕННЫЙ доступ к данным
    products.ts  cart.ts  orders.ts  pricing.ts  ...
  supabase/                # фабрики клиентов (3 файла, не трогать без ADR)
  stripe/                  # клиент + хелперы сессий/вебхуков
  i18n/                    # словари по namespace, конфиг локалей
  money.ts                 # Money-тип и вся денежная арифметика
  events/                  # публикация/подписка доменных событий
components/
  ui/                      # shadcn-примитивы (только используемые)
  <feature>/               # компоненты фичи: cart/, product/, checkout/
supabase/
  migrations/              # только через supabase CLI, нумерованные
  seed.sql
tests/
  e2e/                     # Playwright
  unit/
docs/
  adr/                     # NNNN-title.md
  reports/                 # исторические аудиты
```

**Ответственность:**
- `lib/commerce` — единственное место, где встречаются `createClient()` и SQL-мысли. Функции возвращают доменные типы, не строки таблиц.
- `app/**` — композиция и рендеринг. Никакой бизнес-логики: страница не умеет считать скидку.
- `components/ui` — только реально используемые примитивы shadcn. Добавление компонента = использование в этом же PR.
- `app/api` — тонкие адаптеры: parse → auth/rate-limit → вызов `lib/*` → сериализация. Логика в route-handler — дефект.

## 3. Coding Standards

### TypeScript
- `strict: true`; `ignoreBuildErrors` **MUST NOT** появиться снова. Ошибка компиляции = красный CI = нет merge.
- `as any`, `as never`, `@ts-ignore` — MUST NOT. Исключение — `@ts-expect-error` с причиной и ссылкой на issue.
- Типы БД **MUST** генерироваться (`supabase gen types typescript`); рукописные интерфейсы строк таблиц запрещены — исторически именно они разошлись со схемой.
- Домен моделируется типами: `Money = { amount: number; currency: CurrencyCode }`, `OrderStatus` — union, не `string`.
- Функции `lib/commerce` возвращают Result-типы или бросают типизированные ошибки; «проглотить и вернуть пустой массив» — MUST NOT.

### React
- Компонент > 200 строк — сигнал к декомпозиции; > 400 — блокер ревью (анти-пример: account-страница на 1319 строк).
- Состояние — на минимально достаточном уровне. Новый глобальный контекст — только через ADR; текущий список закрыт: Auth, Cart, Lang.
- Провайдеры монтируются **один раз** в корневом layout. Повторное оборачивание страницы — MUST NOT (исторический дефект: 24 страницы с дублями).
- Побочные эффекты данных — не в `useEffect`. Данные приходят из Server Components / server actions; `useEffect` — для DOM/подписок.

### Next.js App Router
- Страница по умолчанию — RSC. `"use client"` — на листовом интерактивном компоненте, не на странице.
- Каждый публичный маршрут **MUST** объявить стратегию: `static` | `revalidate: N` + cache tags | `dynamic`. «Как получится» — не стратегия.
- Мутации кэша — через `revalidateTag` из событий (товар изменился → tag `product:{id}`).
- Навигация — только `<Link>` / `router.push`. `window.location.href` и голый `<a>` на внутренние маршруты — MUST NOT.
- Middleware выполняет auth-работу **только** на защищённых матчерах (`/account`, `/admin`, `/checkout`), никогда на статике.
- `error.tsx`, `not-found.tsx`, `loading.tsx` обязательны для каждого route-сегмента с данными.

### Server Components
- Читают данные только через `lib/commerce`. Прямой `createClient()` допустим только внутри `lib/`.
- Не передают в клиентские компоненты больше данных, чем те рендерят.
- Секреты и `server-only`-модули не импортируются в файлы с `"use client"` — граница проверяется линтером.

### Client Components
- Причина `"use client"` называется в одну строку: обработчики, состояние, браузерные API. «Так проще» — не причина.
- Не содержат: запросов к БД, денежной арифметики, построения SQL/фильтров, секретов.
- localStorage — только через типизированные хелперы с версией схемы (`pharmiperia:v1:cart`), с try/catch и деградацией.

### Server Actions
- Только для мутаций из форм витрины. Для чтения — RSC; для внешних клиентов — API.
- Каждый action: (1) валидация входа zod, (2) проверка auth/прав, (3) вызов `lib/commerce`, (4) типизированный результат. Всегда в этом порядке.
- Action **MUST NOT** принимать цену, сумму, скидку с клиента. Принимает идентификаторы и количества — суммы считает сервер.
- Идемпотентность денежных операций обязательна (idempotency key).

### API (route handlers)
- Каждый публичный endpoint: rate limit + валидация zod + явный контракт ответа. Endpoint без rate limit — блокер.
- Версионирование с первого внешнего потребителя: `/api/v1/...`.
- Ошибки — структурированные (`{ error: { code, message } }`), без stack traces.

### Supabase
- Три клиента (browser/server/middleware) создаются только фабриками из `lib/supabase`. Четвёртой копии, самодельного парсинга cookie, inline-`createServerClient` — MUST NOT.
- RLS включён на каждой таблице всегда. «Временно без RLS» не существует.
- Новая таблица — в одной миграции с политиками: owner, admin, публичные — осознанно и минимально. Политика, открывающая перечисление чувствительных данных (анти-пример: promo_codes), — блокер.
- Пользовательский ввод **MUST NOT** интерполироваться в `.or()`/фильтры PostgREST строками — только параметризованные методы или RPC.
- `service_role` — только на сервере в `lib/`, никогда в `NEXT_PUBLIC_*`.

### Stripe
- Единственный источник факта оплаты — вебхук с проверкой подписи. UI-колбэки (`onComplete`) — только UX-навигация; MUST NOT менять состояние заказа.
- `line_items` строятся из цен, прочитанных сервером из БД. Комментарий «validated server-side» без валидации — исторический позор проекта; код доказывает, а не заявляет.
- Все суммы — те же минорные единицы из `Money`; конверсия — в одном месте (`lib/stripe`).
- Ключи — restricted keys с минимальными правами. Metadata — только идентификаторы (лимит 500 символов; JSON корзины не кладём).

### SQL / миграции
- Только `supabase/migrations` + CLI; ручные правки схемы в Studio запрещены.
- Миграция = одна тема; копии одного скрипта под разными номерами (исторический дефект) — MUST NOT.
- Каждая FK и частый WHERE — с индексом; `DECIMAL` для денег — запрещён (integer в минорных единицах; существующие колонки — по плану ADR-Money).
- Переводы контента — таблицами переводов, не колонками `_ru`/`_lv` (новые сущности — сразу).

### CSS / Tailwind
- Дизайн-токены — только CSS-переменные из единственного `app/globals.css`. Хардкод цветов (`bg-[#f2f3f5]`, `text-red-500` для семантики) — SHOULD NOT; семантические токены (`bg-muted`, `text-destructive`).
- Условные классы — через `cn()`. Тернарники длиной в абзац — рефакторить в `cva`.
- Никаких инлайн-`style` для выражаемого утилитами.

### shadcn/ui
- Компонент добавляется генератором и используется в том же PR. «Библиотека на будущее» из 57 файлов — исторический дефект, не возвращается.
- Модификации примитива — в самом файле `components/ui/*` с комментарием-маркером.
- Диалоги/дроверы/меню — только на Radix (фокус-трап и клавиатура бесплатно). Самодельный `<div className="fixed">`-модал — MUST NOT.

## 4. Naming Conventions

- **Файлы:** kebab-case (`product-card.tsx`). Компонент — PascalCase-экспорт из kebab-файла.
- **БД:** snake_case, таблицы во множественном (`order_items`), булевы `is_/has_`. Имена колонок в коде — только из сгенерированных типов.
- **Доменные функции:** глагол + сущность: `getProductBySlug`, `addCartLine`, `applyPromoCode`. Один словарь на домен: у корзины есть `addCartLine` — синонимов не существует (исторический дефект: `addToCart`/`addItem`/фантомный `clearCart`).
- **События:** `сущность.глагол` — `order.paid`, `product.updated`, `cart.abandoned`.
- **Env:** `NEXT_PUBLIC_*` — только то, что можно напечатать на футболке. Всё — в `.env.example` (обязателен, актуализируется в том же PR).
- **Переводы:** ключи по namespace: `checkout.shipping.title`.
- **Слаги:** одна функция в `lib/` (нормализация с транслитерацией диакритики). Вторая реализация slug-логики — блокер (исторический дефект: 4 копии → битый sitemap).

## 5. Definition of Done

1. CI зелёный: typecheck, lint, build, тесты.
2. Данные через `lib/commerce`; клиент не получил новых прав.
3. Ошибки обработаны: пользователь видит осмысленное состояние, Sentry видит исключение, «пусто» ≠ «сломалось».
4. Обе локали (lv/ru) через i18n; проверено на обоих языках.
5. Мобильная вёрстка (375px), клавиатурная навигация.
6. SEO для новых страниц: title, description, canonical, стратегия кэширования объявлена.
7. Аналитические события добавлены, если фича — часть воронки.
8. E2E обновлён, если фича касается critical path (каталог→корзина→оплата→заказ).
9. Миграции через CLI, применимы к пустой БД.
10. Мёртвого кода не осталось; `.env.example` и docs актуальны.
11. Staging + тестовый платёж Stripe, если фича касается денег.

## 6. Pull Request Checklist

- [ ] PR ≤ ~400 строк диффа (больше — делить; исключения обоснованы).
- [ ] Описание: что, зачем, как проверить; скриншоты для UI (light, mobile).
- [ ] Нет `console.log`, закомментированного кода, TODO без issue.
- [ ] Нет новых зависимостей без обоснования.
- [ ] Нет `as any` / `@ts-ignore`; нет дублей существующих утилит (искал перед тем как писать).
- [ ] Деньги — только `Money`; даты — UTC ISO в данных.
- [ ] Миграции обратимы или план отката описан.
- [ ] Секреты не в диффе (глазами + secret-scanner в CI).
- [ ] Локальный e2e-смоук, если тронут checkout/cart/auth.

## 7. Code Review Checklist

Порядок — сверху дороже всего пропустить:

1. **Границы:** нет запросов к БД/Stripe из компонентов; нет доверия клиентским суммам/ролям.
2. **Деньги и заказ:** идемпотентность, вебхук-инварианты, отсутствие клиентских переходов статуса.
3. **RLS/доступ:** новая таблица — политики; новый endpoint — auth + rate limit; нет расширения публичной поверхности.
4. **Контракты:** вызываемые методы существуют и семантически верны.
5. **Ошибки:** каждый `catch` обрабатывает или пробрасывает; глотание — блокер.
6. **Кэш/рендеринг:** стратегия объявлена; статичная страница не стала динамической случайно.
7. **i18n/a11y/SEO:** хардкод-строки, фокус, canonical.
8. **Простота:** можно ли меньшим кодом или существующей утилитой.

Тон: комментируем код, не человека; блокеры — `blocking:`, мнения — `nit:`.

## 8. Performance Standards

**Бюджеты (staging, mobile 4G, p75):**
- LCP < 2.5 s, INP < 200 ms, CLS < 0.1; регресс двух недель подряд = инцидент.
- First-load JS страницы витрины < 170 KB gzip; превышение — bundle-analyzer в PR + обоснование.
- TTFB каталожных страниц < 200 ms (ISR-кэш-хит; больше — страница случайно динамическая).

**Правила:**
- Только `next/image` с оптимизацией; `unoptimized` запрещён в проде. Товарные медиа — CDN/storage с `sizes`, hero — `priority`.
- Каталог/товар/категории — ISR + tags; персональное — dynamic; контент — static.
- Ничего below-the-fold не грузится eagerly (`next/dynamic` для чатов/виджетов).
- Словари i18n — по namespace/locale, не монолитом.
- Композиция данных в RSC, не водопад `useEffect`.

## 9. SEO Standards

- Каждая индексируемая страница: уникальные title/description через `generateMetadata`, **собственный canonical** (наследование с корня — исторический дефект, запрещено).
- hreflang-пары lv/ru для каждой публичной страницы; локаль в URL — единственный источник языка для краулера.
- Структурированные данные обязательны: Product+Offer (+AggregateRating), BreadcrumbList, Organization; валидация в e2e.
- URL: слаги, нижний регистр, без ID; изменение — только 301 + запись в реестре редиректов.
- Sitemap из БД (не из констант), шарды по 45k URL; robots закрывает `/account`, `/checkout`, `/api`, `/admin`, параметрические выдачи.
- Новая страница объявляет: индексировать или `noindex` — решение в PR.
- Контент рендерится сервером. Клиентская страница с SEO-контентом — блокер.

## 10. Security Standards

- **Не доверяем клиенту:** цены, суммы, скидки, роли, `user_id` — сервер. Клиент присылает намерения, не факты.
- Вебхуки — проверка подписи + идемпотентность; секреты ротируются по runbook.
- Каждый публичный endpoint — rate limit; auth-поверхности — captcha.
- Заголовки (CSP, HSTS, XFO, XCTO, Referrer-Policy, Permissions-Policy) — централизованно; ослабление CSP — ADR.
- PII: не в localStorage (исторический дефект), не в логах, не в аналитике; PII-поля новых таблиц перечисляются в PR явно.
- `pnpm audit` в CI; новая зависимость — обоснование.
- Секреты — только env; попадание в git = немедленная ротация, не «удаление коммита».
- AI-endpoints: авторизация/лимит, бюджет-кап, ограничение длины ввода в промпт.
- Всё, что видит anon-роль Supabase, считается опубликованным в интернете.

## 11. Accessibility Standards

Цель — WCAG 2.1 AA (EAA действует в ЕС; соответствуем не из-за исключений, а потому что это рынок).

- Любой оверлей — Radix-примитив: фокус-трап, Escape, `aria-modal`, возврат фокуса. Самодельные — запрещены.
- Всё интерактивное достижимо с клавиатуры; hover-only навигация (исторический дефект меню) — запрещена.
- Икон-кнопки — `aria-label`; статусы/тосты — `aria-live`; формы — label + `aria-describedby` для ошибок.
- Контраст текста ≥ 4.5:1 (на токенах); информация не кодируется только цветом.
- `<html lang>` соответствует локали URL.
- axe в e2e на ключевых страницах; новые serious/critical — блокер.

## 12. Testing Standards

- **E2E (Playwright) — обязательный минимум:** happy path покупки (гость и юзер), auth-цикл, сценарии «покупка невозможна». В CI на preview + по крону на проде (тестовая карта).
- **Unit — обязательно для:** `lib/money`, доставки, скидок, cart-логики, slug/i18n-утилит. Деньги без тестов не мержатся.
- **Integration:** `lib/commerce` против локального Supabase — контракт с RLS («anon не может прочитать чужой заказ» так же важно, как «owner может»).
- Тесты — часть PR фичи. Флаки-тест чинится или удаляется за неделю, не skip-ается навечно.
- Метрика — не покрытие, а «critical path не может сломаться незаметно».

## 13. Git Workflow

- **Trunk-based:** `main` защищён (CI + 1 ревью), короткоживущие ветки `feat/…`, `fix/…`, squash-merge.
- **Conventional Commits:** `feat(checkout): …`, `fix(cart): …`.
- `main` всегда деплоибелен; недоделанное — за feature-флагом, не в долгоживущей ветке.
- Миграции БД — отдельным PR раньше зависимого кода (совместимость N и N−1).
- Revert — первая реакция на сломанный прод; разбор — после отката.
- Прямые пуши в `main`, force-push в общие ветки, коммиты из Vercel/v0 UI — запрещены. Единственный источник кода — этот репозиторий.

## 14. Правила работы с AI (Claude, v0, ChatGPT и др.)

AI — ускоритель, автор — ты. История проекта показала, чем кончается обратное.

1. **Ты — автор каждой строки.** Сгенерированный код проходит то же ревью; «это AI сгенерировал» — не объяснение.
2. **Сначала контекст, потом генерация.** AI получает этот Playbook (§1–4, §15) и сигнатуры существующих утилит. Генерация «с чистого листа» — источник четырёх копий `getBrandSlug`; запрещено.
3. **AI не изобретает контракты.** Не проси «предположить» интерфейс — дай реальный файл типов (анти-пример: `clearCart`-синдром).
4. **Никаких артефактов генерации:** `[v0]`-логи, комментарии-пересказы, комментарии-алиби («would come from CMS in real app»), скаффолд целиком — вычищаются до коммита.
5. **Один PR — один автор-человек,** способный объяснить каждое решение в диффе.
6. **AI не мержит без человека:** миграции, RLS, платёжный код, криптографию, конфиг безопасности — построчный человеческий проход.
7. **Не вставляй секреты и PII в промпты** внешних инструментов.
8. **Дубликаты — тест на дисциплину:** перед принятием утилиты — grep на существующую. Вторая реализация — блокер.
9. Большие AI-рефакторинги — через план в issue до генерации.

## 15. Нерушимые архитектурные решения

Нарушение — блокер без исключений; изменение — только ADR + одобрение tech lead:

1. Деньги — минорные единицы + код валюты, всегда.
2. Цены/суммы/статусы заказов определяет сервер; вебхук Stripe — единственный источник факта оплаты.
3. Один источник данных о товарах — Postgres/Supabase.
4. Доступ к данным — только через `lib/commerce`; компоненты не импортируют Supabase-клиент.
5. RLS включён на каждой таблице; anon-поверхность = публичная.
6. Локаль и слаги — в URL; опубликованные URL меняются только через 301.
7. `strict` TypeScript и зелёный CI — условие merge; отключение проверок сборки запрещено.
8. Провайдеры контекста — один раз в корне.
9. Секреты не покидают сервер; `NEXT_PUBLIC_*` — только публичное по смыслу.
10. Побочные эффекты заказа — через события, не клиентские колбэки.
11. Оверлеи — на доступных примитивах.
12. Миграции — только через CLI-конвейер.

## 16. Что требует ADR

ADR (`docs/adr/NNNN-title.md`: контекст → решение → последствия → альтернативы) обязателен для:

- Изменения схемы URL, структуры локалей, доменной стратегии.
- Изменений схемы денег, налогов, валют; смены платёжного провайдера или flow.
- Новых таблиц ядра (orders, products, carts), изменения их ключей/связей; смены модели переводов.
- Нового глобального состояния, нового рантайма (edge), нового внешнего сервиса (поиск, очередь, CMS, аналитика, AI-провайдер).
- Изменения стратегии кэширования класса страниц; ослабления безопасностных правил.
- Введения публичного API/версии; всего, что будет использовать мобильное приложение.
- Отступления от любого пункта §15.

Правило большого пальца: если через год «почему здесь так?» не очевидно из кода — нужен ADR.

## 17. Ошибки, которые запрещено повторять

Выстраданный список — каждый пункт был в проекте реально:

1. **Фантомный контракт** — вызов несуществующего метода (`clearCart`, `addItem`) под отключённым typecheck'ом. Урок: CI-гейт не отключается «на время».
2. **Комментарий-ложь** — «prices validated server-side» без валидации.
3. **Двойной источник истины** — каталог в TS рядом с каталогом в БД; запросы «под воображаемую схему».
4. **Копипаст утилиты** — четыре `getBrandSlug` → битые URL в sitemap.
5. **Провайдер поверх провайдера** — 24 страницы с расщеплённым состоянием.
6. **Успех без подтверждения** — «спасибо за заказ» при упавшей записи; `catch { console.error }` на деньгах.
7. **Клиент решает про деньги** — цена из браузера, статус `paid` из колбэка.
8. **Публичная поверхность по недосмотру** — RLS, отдающая промокоды анониму; открытые AI-endpoints.
9. **PII в localStorage** — профиль с телефоном и адресом в браузере.
10. **Фейковые фичи** — форма связи с `setTimeout` и ложным «отправлено». Фича либо работает, либо не показывается.
11. **Canonical на всё** — один canonical в корневом layout на весь сайт.
12. **Склад ненужного** — 57 ui-компонентов, мёртвые компоненты, дубли SQL, неиспользуемые зависимости.
13. **Хардкод чужого языка** — латышский/русский/английский мимо i18n.
14. **Самодельный модал** — оверлей без фокус-трапа и Escape.
15. **`window.location.href` внутри SPA** — полная перезагрузка на каждый поиск.
16. **Запуск на незарегистрированном домене** — canonical, sitemap и почта, указывающие на домен, которым никто не владеет. Инфраструктурные предпосылки проверяются до кода, который на них ссылается.
